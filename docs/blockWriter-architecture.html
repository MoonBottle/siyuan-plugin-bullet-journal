BlockWriter 架构说明

SiYuan 任务助手插件 · 块写入引擎 · B/C 阶段架构

核心调用链

writeBlock() / insertBlockAfter() → normalizeIntent → buildMutationPlans → executePlans → commit
1
入口层 index.ts
writeBlock(context, patches) 或 insertBlockAfter(blockId, patch)， 薄入口，只做参数转发，不做业务判断
2
意图归一化 intent/intent.ts + normalizePatchSequence.ts
将裸 patch 数组归一为 BlockMutationIntent，按 PATCH_ORDER 稳定排序， 确保不同入口表达同一业务语义时，进入 planner 前的 patch 序列一致
3
规划器 planner/mutationPlanner.ts
5 阶段规划：Normalize → Capability Annotation → Merge Attempt → Minimal Split → Order Resolution
输出 MutationPlannerResult，包含执行计划列表和拆分原因
4
执行引擎 runtime/mutationExecutor.ts
按 plan 顺序执行，每个 plan 内：load source → render payload → commit
protyle commit 失败时自动 fallback 到 API
5
提交层 commit/protyleCommitter.ts + apiCommitter.ts
Protyle 提交：transaction + 光标恢复（wbr-first, offset 兜底）
API 提交：DOM-first，markdown fallback

Planner 5 阶段详解

Phase 1: Normalize

将裸 patch 数组归一为 MutationPatchUnit[]，保留原始顺序，标记 intentKind。

Phase 2: Capability Annotation

对每个 unit 调用 resolveMutationTarget，得到能力描述 MutationPatchCapability：

字段 含义
targetBlockId 真实目标块 ID
sourceKind protyle-dom 或 api-kramdown
commitKind protyle-update 或 api-update
preferredCaretPolicy slash cleanup → wbr，其他 → none

Phase 3: Merge Attempt

按顺序遍历 capability，判断是否可与前一个合并。合并条件：

intentKind 相同 && targetBlockId 相同 && sourceKind 相同 && sourceBlockId 相同 && commitKind 相同

合并后 caretPolicy 取更强策略：wbr > none

Phase 4: Minimal Split

无法合并时，按优先级确定拆分原因：

优先级 拆分原因 场景
1 split-by-intent-kind update 与 insertAfter 混合
2 split-by-target 不同真实目标块
3 split-by-source protyle-dom 与 api-kramdown 混合
4 split-by-commit-kind protyle-update 与 api-update 混合

Phase 5: Order Resolution

从后往前找最后一个 protyle-dom plan，标记为 caretOwner。只有 caretOwner 负责 最终光标恢复，避免多个 plan 竞争恢复光标。

执行模型

单 Plan 执行（理想路径）

loadMutationSource → prepareUpdatePayload → commitViaProtyle 或 commitViaApi

多 Plan 执行

  • 按 plan order 顺序执行
  • 每个 plan 内只提交一次
  • 默认 fail-fast
  • 不做跨 plan 伪事务回滚
  • 只有 caretOwner === true 的 plan 负责光标恢复

API Fallback

当 protyle commit 失败时，executor 自动使用 apiFallbackPlan 中的信息重新加载 source、 重新渲染 payload、通过 API 提交。

Patch 类型一览

类型 用途 排序权重
removeSlashCommand 删除 slash 命令文本 0
setContent 设置事项内容 10
addDate 添加/修改日期 20
setReminder 设置提醒 30
setRecurring 设置重复规则 40
setPriority 设置优先级 50
setTaskTag 设置任务标签 55
setFocusPlan 设置专注计划 60
togglePinned 切换置顶 70
setHabitArchive 设置习惯归档 80
setStatus 设置状态 90
setHabitDefinition 设置习惯定义 100
setHabitRecord 设置习惯记录 110
replaceMarkdown 替换整个 markdown 120
排序设计决策：removeSlashCommand 最先执行（清理 slash 文本后才能正确渲染后续内容）， setStatus 靠后（状态变更通常伴随内容变更，需要先完成内容渲染）， replaceMarkdown 最后（它是全量替换，会覆盖前面所有变更）。

光标恢复策略

caretOwner plan → wbr 主恢复 成功？ 完成 / 失败 → offset 兜底
  • wbr 主恢复：在渲染 DOM 时注入 <wbr> 标记，commit 后通过 focusByWbr() 定位
  • offset 兜底：记录 commit 前的光标偏移量，wbr 丢失时通过 focusByOffset() 恢复
  • caretOwner：只有最后一个 protyle-dom plan 被标记为 caretOwner，避免多 plan 竞争恢复

合并与拆分规则

可合并（优先合并为单 plan）

场景 条件
removeSlashCommand + setStatus 同 target
removeSlashCommand + addDate 同 target
removeSlashCommand + setContent 同 target
setPriority + setContent + addDate 同 target / source / commit
任意同目标、同 source、同 commit 的 patch 序列 能力一致

不可合并（必须拆分）

场景 拆分原因
slash cleanup 目标 ≠ 业务更新目标 split-by-target
部分需 protyle-dom，部分只能 api-kramdown split-by-source
一个是 insertAfter，一个是 update split-by-intent-kind
真实目标块不同 split-by-target
