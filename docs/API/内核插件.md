内核侧 JavaScript 插件运行时
概述
本 PR 为思源笔记引入了内核侧 JavaScript 插件运行时，允许插件通过 goja 引擎在内核进程内直接运行 JavaScript（kernel.js）。插件可访问一个沙箱化的 siyuan 全局对象，该对象暴露了生命周期钩子、结构化日志、作用域文件存储、JSON-RPC 服务器、HTTP fetch、双向 WebSocket 支持以及 Server-Sent Events——全部基于镜像浏览器约定的 Promise 异步 API。

架构
内核进程
└── PluginManager（单例）
    └── KernelPlugin（每个已启用的插件一个）
        ├── Worker          — 在 goja 事件循环上序列化 Go↔JS 调用
        ├── goja EventLoop  — 每个插件独立的单线程 JS 运行时
        ├── EventBus        — 进程内发布/订阅（运行时 / 插件主题）
        ├── rpcMethods      — 已注册 JSON-RPC 方法的 sync.Map
        └── sockets         — 已追踪的 WebSocket 连接（客户端 + 服务端）
每个 KernelPlugin 拥有一个独立的 goja 事件循环，因此插件之间不会相互干扰，也不会阻塞内核的 goroutine。

插件加载
PluginManager.Start() 调用 model.LoadKernelPetals()，该函数遍历所有已启用的 petal，并在以内核插件方式启动之前按顺序进行以下检查。

1. plugin.json — kernels 字段
plugin.json 清单文件中必须包含非空的 kernels 数组。该字段同时承担两个作用：将插件纳入内核插件支持范围，同时声明所支持的平台。若 kernels 字段缺失或为空，则不加载内核插件，加载器会静默跳过。

{
  "kernels": ["windows", "linux", "darwin"]
}
可接受的平台值："windows"、"linux"、"darwin"、"docker"、"android"、"ios" 或 "all"。

若当前平台（桌面构建使用 runtime.GOOS，否则使用容器标识符）未在 kernels 中列出，则 Kernel.Incompatible 被设为 true，插件被跳过。

2. kernel.js 存在性检查
通过 kernels 检查后，loadCode 会查找 <data>/plugins/<name>/kernel.js。若文件存在，则读入内存（Petal.Kernel.JS）并将 Petal.Kernel.Existed 设为 true。若文件不存在，加载器记录警告日志并跳过该内核插件的启动：

[WARN] plugin [my-plugin] kernel.js not found, skip loading as kernel plugin
决策流程
已启用?           → 否 → 跳过
kernels 存在?     → 否 → 跳过（Kernel.Incompatible = true）
kernels 匹配?     → 否 → 跳过（Kernel.Incompatible = true）
kernel.js 存在?   → 否 → 跳过（警告）
                    是 → KernelPlugin.start()
文件	用途
kernel/plugin/plugin.go	KernelPlugin 结构体、生命周期（start / stop）、钩子调用、RPC 分发、服务端 WS/SSE 请求处理
kernel/plugin/manager.go	PluginManager 单例——并发地发现、启动和停止插件
kernel/plugin/sandbox.go	EnableSiyuanModule——向每个 JS 运行时注入完整的 siyuan.* API
kernel/plugin/api_client.go	注入 siyuan.client——HTTP fetch、WebSocket 客户端、SSE/EventSource 客户端
kernel/plugin/api_server.go	注入 siyuan.server——HTTP/WebSocket/SSE 服务端处理器插槽
kernel/plugin/websocket.go	WsEventHandler——将 gws WebSocket 事件桥接到 JS 运行时（客户端与服务端均使用）
kernel/plugin/rpc.go	JSON-RPC 2.0 HTTP + WebSocket 处理器，广播通知支持
kernel/plugin/worker.go	Worker——对 goja 事件循环的轻量封装，用于异步任务调度
kernel/plugin/server.go	Request / HttpResponse 类型、请求解析、HandleHttpRequest 路由

await siyuan.logger.trace(...args);

console.log("hello", 42);       // → 内核 INFO 日志
console.info("hello", 42);      // → 内核 INFO 日志
console.warn("something odd");  // → 内核 WARN 日志
console.error("oops", err);     // → 内核 ERROR 日志

console 方法	内核级别

log	INFO
info	INFO

await siyuan.rpc.bind("echo", async (...args) => args, "description");

siyuan.client — 网络客户端工具（已冻结）
提供 HTTP、WebSocket 和 SSE 访问，全部经由内核隧道转发，并自动注入插件 JWT token 进行鉴权。

siyuan.client.fetch — 隧道 HTTP
const resp = await siyuan.client.fetch("/api/system/version", { method: "POST", body: "{}" });
resp.ok;          // boolean
resp.status;      // number
resp.statusText;  // string
resp.headers;     // Record<string, string>
await resp.text();          // Promise<string>
await resp.json();          // Promise<any>
await resp.arrayBuffer();   // Promise<ArrayBuffer>
请求发送至 http://127.0.0.1:<port><path>，插件的 JWT token 由内核自动注入。

siyuan.client.socket — WebSocket 客户端
立即返回一个处于 CONNECTING 状态的已密封 IWebSocket 对象，必须显式调用 ws.open() 才能发起 TCP/WebSocket 握手——只有在 open() 完成后，onopen 才会触发。多次调用 open() 是幂等的。

const ws = await siyuan.client.socket("/ws/plugin/rpc/my-plugin");
// 先绑定回调，再调用 open()：
ws.onopen    = async (event) => {
    await ws.ping("ping");
    await ws.send("hello");
};
ws.onmessage = (event) => { /* event.data: string | ArrayBuffer */ };
ws.onping    = (event) => { };
ws.onpong    = (event) => { };
ws.onerror   = (event) => { };
ws.onclose   = (event) => { };
// 发起连接：
await ws.open();
// 其他方法：
await ws.ping("data");
await ws.pong("data");
await ws.close(1000, "bye");
socket 对象的只读属性：

属性	类型	说明
binaryType	string	始终为 "arraybuffer"
bufferedAmount	number	已入队待发送但尚未传输的字节数
extensions	string	已协商的 WebSocket 扩展，若无则为 ""
protocol	string	已协商的子协议，若无则为 ""
readyState	number	0 CONNECTING · 1 OPEN · 2 CLOSING · 3 CLOSED
url	string	完整 WebSocket URL，例如 "ws://127.0.0.1:6806/ws/…"
消息事件说明： onmessage 事件中，文本帧携带 { type: "text", data: string }，二进制帧携带 { type: "binary", data: ArrayBuffer }。

siyuan.client.event — SSE / EventSource 客户端
打开一个 Server-Sent Events 流。立即返回已密封的 IEventSource 对象，内核在后台启动 SSE 订阅。

const es = await siyuan.client.event("/es/broadcast/subscribe");
es.onopen    = async (event) => { await siyuan.logger.debug("SSE open", event); };
es.onmessage = async (event) => {
    // event.type: SSE event: 字段（默认为 "message"）
    // event.data: SSE data: 字段（UTF-8 字符串）
    // event.lastEventId: SSE id: 字段
    await siyuan.logger.debug("SSE message", event);
};
es.onclose   = async (event) => { await siyuan.logger.debug("SSE close", event); };
es.onerror   = async (event) => { await siyuan.logger.debug("SSE error", event); };
// 取消订阅：
es.close();
siyuan.server — Web 请求处理器注册表（已冻结）
/plugin/private/<name>/*path 下的路由需通过内核鉴权（admin 角色）。为各插槽的 .handler 属性赋值函数即可注册处理器，内核在分发请求后会等待返回的 Promise 完成。

三种处理器类型均接收相同的基础 IServerRequest 对象（request.url、request.request、request.context）。WebSocket 和 SSE 处理器还额外接收 request.port——与已连接客户端通信的信道。

HTTP 处理器 — siyuan.server.private.http.handler
接收完整的 IServerRequest，必须返回 IHttpResponse（或其 Promise）。

siyuan.server.private.http.handler = async (request) => {
    await siyuan.logger.debug("HTTP 请求", request.url.path);
    return {
        statusCode: 200,
        headers: { "X-Plugin": ["my-plugin"] },
        body: {
            data: { type: "JSON", data: { ok: true } },
        },
    };
};
IHttpResponse.body 支持以下几种形式：data（序列化数据——JSON/XML/YAML/…）、file（本地文件系统）、string（fmt.Sprintf 格式化字符串）、raw（字节数据 + Content-Type）或 redirect（重定向）。省略 body 则返回仅含状态码的响应。

WebSocket 服务端处理器 — siyuan.server.private.ws.handler
接收 { url, request, context, port }。port 对象与 IWebSocket 接口相同（相同的事件回调、相同的 send/ping/pong/close 方法）。内核在调用处理器前已完成 WebSocket 升级；处理器返回后，若未显式调用 port.open()，内核会自动启动读取循环。

siyuan.server.private.ws.handler = async (request) => {
    request.port.onopen = async (event) => {
        await request.port.send("来自插件的问候！");
    };
    request.port.onmessage = async (event) => {
        // event.type: "text" | "binary"
        // event.data: string | ArrayBuffer
        await request.port.send(event.data);   // 回显
    };
    request.port.onping = async (event) => {
        await request.port.pong(event.data);
    };
    request.port.onclose = async (event) => {
        await siyuan.logger.debug("客户端断开连接", event);
    };
    request.port.onerror = async (event) => {
        await siyuan.logger.debug("ws 错误", event.error);
    };
    // port.open() 可选——处理器返回后内核会自动打开。
};
SSE 服务端处理器 — siyuan.server.private.es.handler
接收 { url, request, context, port }。port 提供同步的 send 和 close 方法，以及 onopen/onclose 回调。处理器返回后内核开始流式传输；port.onopen 在流式传输开始时触发。连接保持开启直至调用 port.close() 或客户端断开连接。

siyuan.server.private.es.handler = async (request) => {
    request.port.onopen = async (event) => {
        // send(eventType, data) — eventType 对应 SSE 的 event: 字段
        request.port.send("message", "已连接到插件 SSE！");
        request.port.send("update", JSON.stringify({ ts: Date.now() }));
    };
    request.port.onclose = async (event) => {
        await siyuan.logger.debug("SSE 客户端断开连接");
    };
};

HTTP 方式的 JSON-RPC 2.0
方法	路径
POST	/api/plugin/rpc/:name
POST	/api/plugin/rpc?name=my-plugin
单请求
标准 JSON-RPC 2.0 调用。params 字段可以是位置数组或命名对象。

参考：JSON-RPC 2.0 规范

{ "jsonrpc": "2.0", "method": "echo", "params": [1, "hello"], "id": 1 }
响应

{ "jsonrpc": "2.0", "result": [1, "hello"], "id": 1 }
通知
通知是不含 id 成员的请求对象。规范规定："作为通知的请求对象表示客户端对相应响应对象不感兴趣，因此无需向客户端返回响应对象。服务器不得回复通知。"

服务器返回 204 No Content。错误（无效参数、内部错误）会被静默丢弃——客户端不会收到通知。

{ "jsonrpc": "2.0", "method": "echo", "params": ["fire-and-forget"] }
批量请求
规范规定："客户端可以发送一个填充了请求对象的数组。服务器可以将批量 RPC 调用作为一组并发任务处理，以任意顺序和任意并行度进行处理。"

实现说明：本服务器通过 goroutine 并发分发所有批量项，并在写入前收集所有响应。

请求

[
  { "jsonrpc": "2.0", "method": "echo", "params": ["a"], "id": 1 },
  { "jsonrpc": "2.0", "method": "echo", "params": ["b"], "id": 2 },
  { "jsonrpc": "2.0", "method": "echo", "params": ["notify-only"] }
]
响应 — 不保证顺序的响应数组；使用 id 进行匹配。规范规定："通知不应有对应的响应对象。" 若所有项均为通知，服务器返回 204 No Content（规范："服务器不得返回空数组"）。

[
  { "jsonrpc": "2.0", "result": ["b"], "id": 2 },
  { "jsonrpc": "2.0", "result": ["a"], "id": 1 }
]
错误响应
{ "jsonrpc": "2.0", "error": { "code": -32601, "message": "Method not found" }, "id": 1 }
预定义错误码（参照规范）：

错误码	消息	含义
-32700	Parse error	收到无效 JSON
-32600	Invalid Request	非有效请求对象
-32601	Method not found	方法不存在或不可用
-32602	Invalid params	无效的方法参数
-32603	Internal error	JSON-RPC 内部错误
-32002	Plugin not running	插件已加载但未处于 running 状态
-32001	Plugin not loaded	未找到该名称的已加载插件
WebSocket 方式的 JSON-RPC 2.0
方法	路径
GET（WS 升级）	/ws/plugin/rpc/:name
GET（WS 升级）	/ws/plugin/rpc?name=my-plugin
帧格式与 HTTP 完全相同——单请求、批量请求和通知均在同一文本帧中支持。服务器连接保持开启以支持全双工通信。

客户端 → 服务器
以文本 WebSocket 消息发送任意有效的 JSON-RPC 2.0 请求帧（单个或批量）。服务器在帧内回复，通过 id 进行匹配。通知不会收到回复帧。

服务器 → 客户端（推送通知）
插件调用 siyuan.rpc.broadcast(method, params) 向所有已连接的 RPC WebSocket 客户端推送未经请求的通知。传输格式为 JSON-RPC 2.0 通知（不含 id）：

{ "jsonrpc": "2.0", "method": "unload", "params": ["Plugin is unloading"] }
规范规定，通知不含 id 字段，因此客户端不得期望这些帧的响应——它们仅为服务器主动发起的事件。

正向代理 API
三个配套端点允许 kernel.js（或任意已授权客户端）通过内核访问外部 HTTP、WebSocket 和 SSE 服务，从而绕过浏览器的 CORS 限制，同时保持在内核 SSRF 安全拨号器的保护范围内。

三个端点均需要 CheckAuth + CheckAdminRole，共享以下两个查询参数：

参数	格式	说明
u	目标 URL 的 RawURLEncoding base64	必填。scheme 须与端点匹配（见下文）。
h	JSON Record<string, string[]> 的 RawURLEncoding base64	可选。转发给目标的附加请求头。
目标响应头会以 Siyuan-Proxy- 为前缀复制到客户端响应中（例如 Content-Type → Siyuan-Proxy-Content-Type）。

所有连接均使用 SSRF 安全拨号器（超时 30 秒），会阻止对私有/回环地址的请求。

HTTP 代理
方法	路径
ANY	/api/network/proxy
将请求（方法 + 请求体 + 可选头部）转发至目标 http:// 或 https:// URL，并将响应原样流回客户端。

GET /api/network/proxy
    ?u=<base64(https://example.com/api/data)>
    &h=<base64({"Authorization":["Bearer token"]})>
WebSocket 代理
方法	路径
GET（WS 升级）	/ws/network/proxy
拨号至目标 ws:// 或 wss:// URL，在浏览器客户端与目标之间双向中继帧。目标握手响应头（加 Siyuan-Proxy- 前缀）作为升级响应头转发给客户端。

GET /ws/network/proxy
    ?u=<base64(wss://example.com/stream)>
    &h=<base64({"Sec-WebSocket-Protocol":["chat"]})>
SSE / EventSource 代理
方法	路径
GET	/es/network/proxy
打开目标 http:// 或 https:// URL 的 SSE 流并管道传输给客户端。若请求头中未设置 Accept，则自动补充 Accept: text/event-stream。

GET /es/network/proxy
    ?u=<base64(https://example.com/events)>
在 kernel.js 中的用法：

内核运行时集成了兼容 Node.js 的 Buffer API，可直接使用 base64Url (⚠️ 注意这里不是 base64url) 编码：

const b64 = (s) => Buffer.from(s).toString("base64Url");

// HTTP 代理
const resp = await siyuan.client.fetch(
    `/api/network/proxy?u=${b64("https://example.com/api/data")}`,
    { method: "GET" }
);

// WebSocket 代理
const ws = await siyuan.client.socket(
    `/ws/network/proxy?u=${b64("wss://example.com/stream")}`
);
await ws.open();

// SSE 代理
const es = await siyuan.client.event(
    `/es/network/proxy?u=${b64("https://example.com/events")}`
);

插件生命周期状态
ready -> loading --> loaded --> running --> stopping --> stopped
                  ↘ onload  ↘ onloaded ↘ onrunning ↘ onunload
PluginManager 在启动时并发启动所有符合条件的内核插件，在关闭时并发停止它们。单个插件可通过 SetPetalEnabled 按需启动/停止。

状态转换与钩子时序
状态	进入条件	调用的钩子
loading	start() 开始执行；JS 运行时已初始化	onload
loaded	onload 返回/解析完成	onloaded
running	onloaded 返回/解析完成	onrunning
stopping	stop() 开始执行	onunload
stopped	onunload 返回/解析完成；socket 已关闭，RPC 方法已清空	—
error	启动过程中发生未恢复的 panic	—
钩子阻塞语义
每次钩子调用从内核视角来看是同步的：生命周期状态机在钩子返回的 Promise 解析之前（或钩子同步返回之前）不会推进到下一状态。内核为每次调用分配一个 Go channel，并阻塞直到 JS Promise 完成：

内核 goroutine            goja 事件循环
      │                         │
      │── worker.Run ──────────>│  调用钩子函数
      │                         │  ← 异步：返回 Promise
      │<── (任务已入队) ─────────│
      │                         │  Promise 解析
      │<── done ← result ───────│
      │  推进状态                │
这意味着缓慢或永不解析的 async 钩子将阻塞插件自身的启动/关闭流程。钩子应保持非阻塞；任何耗时较长的工作应延后执行（例如在 onrunning 中启动但不等待其完成）。

RPC 调用仅在 running 状态下被接受
HTTP 处理器（POST /api/plugin/rpc/:name）和 WebSocket 处理器（GET /ws/plugin/rpc/:name）在插件未处于 running 状态时，均会以 JSON-RPC 内部错误拒绝请求：

{ "jsonrpc": "2.0", "error": { "code": -32002, "message": "Plugin not running" }, "id": 1 }
这意味着：

在 onrunning 返回之前发起的调用将被拒绝——siyuan.rpc.bind 应在 onload 中调用，但调用方必须等到插件进入 running 状态后再发起请求。
在 stop() 发起之后（状态为 stopping 或更后）发起的调用同样会被拒绝。
siyuan.rpc.broadcast（服务器推送）遵循相同的守卫逻辑：若插件不处于 running 状态，则为空操作。
示例 kernel.js
一个可运行的示例，覆盖完整的 API 接口：生命周期钩子、存储 CRUD、RPC 方法注册、HTTP fetch、WebSocket 客户端、SSE 客户端、事件处理、服务端 HTTP/WebSocket/SSE 处理器以及 rpc.broadcast。

class Plugin {
    siyuan = globalThis.siyuan;
    ws = null;
    es = null;

    constructor() {
        this.siyuan.plugin.lifecycle.onload    = this.onload.bind(this);
        this.siyuan.plugin.lifecycle.onloaded  = this.onloaded.bind(this);
        this.siyuan.plugin.lifecycle.onrunning = this.onrunning.bind(this);
        this.siyuan.plugin.lifecycle.onunload  = this.onunload.bind(this);

        this.siyuan.event.handler = this.eventHandler.bind(this);

        this.siyuan.server.private.http.handler = this.httpHandler.bind(this);
        this.siyuan.server.private.ws.handler   = this.wsHandler.bind(this);
        this.siyuan.server.private.es.handler   = this.esHandler.bind(this);
    }

    async onload() {
        // 提前注册 RPC 方法，确保插件进入 running 状态时方法已就绪。
        await this.siyuan.rpc.bind("echo", async (...args) => {
            await this.siyuan.logger.debug("echo called:", args);
            return args;
        }, "原样返回所有接收到的参数。");

        // 存储 CRUD
        await this.siyuan.storage.put("temp", new Date().toISOString());
        const text = await (await this.siyuan.storage.get("temp")).text();
        await this.siyuan.logger.debug("storage 往返测试:", text);
        await this.siyuan.storage.remove("temp");
    }

    async onloaded() {
        // 通过 siyuan.client.fetch 调用内核 REST API
        const resp = await this.siyuan.client.fetch("/api/plugin/listLoadedPlugins", {
            method: "POST",
            body: "{}",
        });
        await this.siyuan.logger.debug("已加载插件:", await resp.json());
    }

    async onrunning() {
        // HTTP RPC 调用（回环：JS → HTTP → Go → JS → Go → HTTP → JS）
        const echoResp = await this.siyuan.client.fetch(
            `/api/plugin/rpc/${this.siyuan.plugin.name}`,
            {
                method: "POST",
                body: JSON.stringify({
                    jsonrpc: "2.0",
                    method: "echo",
                    params: ["hello from fetch", 42],
                    id: 1,
                }),
            }
        );
        await this.siyuan.logger.debug("echo RPC:", await echoResp.json());

        // WebSocket 客户端——先绑定回调，再调用 open() 发起连接
        this.ws = await this.siyuan.client.socket(
            `/ws/plugin/rpc/${this.siyuan.plugin.name}`
        );
        this.ws.onopen = async (event) => {
            await this.siyuan.logger.debug("WebSocket open:", event);
            await this.ws.ping("ping from plugin");
            await this.ws.send(JSON.stringify({
                jsonrpc: "2.0",
                method: "echo",
                params: { message: "hello via WebSocket" },
                id: 2,
            }));
        };
        this.ws.onmessage = (event) => {
            this.siyuan.logger.debug("WebSocket message:", event);
        };
        this.ws.onping  = (e) => { this.siyuan.logger.debug("ws ping:", e); };
        this.ws.onpong  = (e) => { this.siyuan.logger.debug("ws pong:", e); };
        this.ws.onerror = (e) => { this.siyuan.logger.debug("ws error:", e); };
        this.ws.onclose = (e) => { this.siyuan.logger.debug("ws close:", e); };
        await this.ws.open();   // ← 必须调用以发起连接

        // SSE / EventSource 客户端
        this.es = await this.siyuan.client.event("/es/broadcast/subscribe");
        this.es.onopen    = async (e) => { await this.siyuan.logger.debug("SSE open:", e); };
        this.es.onmessage = async (e) => { await this.siyuan.logger.debug("SSE message:", e); };
        this.es.onclose   = async (e) => { await this.siyuan.logger.debug("SSE close:", e); };
        this.es.onerror   = async (e) => { await this.siyuan.logger.debug("SSE error:", e); };
    }

    async onunload() {
        await this.siyuan.rpc.broadcast("unload", ["Plugin is unloading"]);
        this.ws?.close();
        this.es?.close();
    }

    async eventHandler(event) {
        await this.siyuan.logger.debug("内核事件:", event);
        await this.siyuan.event.emit("plugin", { type: "echo", event });
    }

    // HTTP 处理器——返回 IHttpResponse
    async httpHandler(request) {
        return {
            statusCode: 200,
            headers: { "X-Plugin": ["my-plugin"] },
            body: {
                data: { type: "JSON", data: { path: request.url.path } },
            },
        };
    }

    // WebSocket 服务端处理器——request.port 与 IWebSocket 客户端接口相同
    async wsHandler(request) {
        request.port.onopen = async (event) => {
            await this.siyuan.logger.debug("ws port open:", event);
            await request.port.send("来自插件 WebSocket 服务端的问候！");
        };
        request.port.onmessage = async (event) => {
            // event.type: "text" | "binary"
            await request.port.send(event.data);   // 回显
        };
        request.port.onping  = async (e) => { await request.port.pong(e.data); };
        request.port.onclose = async (e) => { await this.siyuan.logger.debug("ws port close:", e); };
        request.port.onerror = async (e) => { await this.siyuan.logger.debug("ws port error:", e); };
        // port.open() 可选——处理器返回后内核会自动打开。
    }

    // SSE 服务端处理器——request.port.send(eventType, data) 推送 SSE 事件
    async esHandler(request) {
        request.port.onopen = async (event) => {
            await this.siyuan.logger.debug("sse port open:", event);
            // send(eventType, data) — eventType 对应 SSE 的 event: 字段
            request.port.send("message", "已连接到插件 SSE！");
        };
        request.port.onclose = async (event) => {
            await this.siyuan.logger.debug("sse port close:", event);
        };
    }
}

const plugin = new Plugin();
