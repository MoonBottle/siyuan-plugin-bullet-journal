数据同步与触发流向

这份文档描述当前实现里的同步主链路：刷新请求如何生成、如何被合并、哪些地方真正刷新数据、 以及 eventBus 与 BroadcastChannel 如何把结果传回各视图。 其中 refreshRequests.ts 负责刷新协议，plugin.requestRefresh() 负责 UI 侧可 await 的 imperative 刷新入口。

主刷新链路

主链路最容易混淆的地方是：projectStore 负责“本地 DATA_REFRESHED”， 而 processRefreshRequest() 负责“刷新完成后的 SETTINGS_CHANGED 与跨上下文广播”。 需要等待刷新稳定完成的手动刷新或交互写操作，应直接 await plugin.requestRefresh(...)。

刷新请求类型

settings-only
directed
full
  • settings-only: 仅设置变更，不要求重扫项目。
  • directed: 附带 docIds，用于局部刷新。
  • full: 全量刷新，常见于 onDataChanged、removeDoc、ws-main 全量命令。

主要触发源

  • UI buttons / interaction flows -> await plugin.requestRefresh(...)
  • onDataChanged() -> full
  • Events.LOCAL_DATA_MUTATED -> directed，失败回退 full
  • ws-main removeDoc -> full
  • ws-main savedoc -> directed
  • ws-main transactions/updateAttrs -> directed
  • SettingsDialog / slashCommands / aiToolsExecutor / pomodoroStore -> submitRefreshRequest

ws-main 分流

全量刷新命令

txerr, refreshdoc, createdailynote, moveDoc

定向刷新命令

savedoc，以及 transactions 中含 updateAttrs 的情况。

rootIDs 提取失败

从 directed 降级为 full，reason 形如 savedoc:missing-rootIDs。

加载路径对照

loadProjects()

  • 首次加载路径
  • 使用 loading = true
  • 显示首次加载态
  • 按批次追加写入 projects

refreshFull()

  • 非首次全量刷新路径
  • 通过 refresh() 进入
  • 使用 refreshing = true
  • 完整构建后一次性替换 projects

refreshDirtyDocs()

定向刷新路径，只重建脏文档对应的数据，避免整份项目列表被全量替换。

本次“首次全量刷新体验”优化只作用于 loadProjects()。非首次全量刷新仍然沿用 refresh() -> refreshFull() 的一次性替换策略。

loading 与 refreshing

loading

  • 表示首次加载中
  • 由 loadProjects() 使用
  • 用于决定是否显示首次加载态

refreshing

  • 表示后台刷新中
  • 由 refresh() 使用
  • 不走首次加载的全屏 loading 逻辑

当前实现里的注意点

  • 同上下文的 DATA_REFRESHED 由 projectStore.loadProjects()/refresh() 发出。
  • 跨上下文的 DATA_REFRESHED 由 processRefreshRequest() -> broadcastDataRefreshed() 发出。
  • SettingsDialog.vue 当前会直接发一次 SETTINGS_CHANGED，同时也会走正式 refresh request 流程，所以设置保存场景存在“即时通知 + 刷新完成通知”。
  • settings-only 不会广播 DATA_REFRESHED。
  • refreshCoordinator 会把连续请求压成串行批次，并在同一批次里用 full > directed > settings-only 规则决定执行路径。
