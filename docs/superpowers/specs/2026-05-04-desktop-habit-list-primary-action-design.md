# Desktop Habit List Primary Action Design

## 背景

当前桌面端 HabitDock 列表项的交互为：

- 点击列表主区域：打开习惯定义所在文档
- 点击右侧日历图标：进入习惯详情视图

这与当前 HabitDock 的主要使用场景不完全一致。HabitDock 本身是围绕“查看打卡状态、进入详情、切换日期、查看统计与日志”构建的工作面板，而不是单纯的文档跳转入口。

## 目标

调整桌面端 HabitDock 列表项的主次操作，使主操作与 HabitDock 的核心任务一致，同时保留快速打开原文档的能力。

## 决策

桌面端 HabitDock 列表项改为：

- **点击主区域**：进入该习惯的详情视图
- **右侧辅助图标**：改为文档图标，点击打开习惯定义所在文档
- **右侧打卡按钮**：保持现有直接打卡/递增行为不变

## 交互定义

### 列表项主区域

- 点击习惯名、连续天数、进度条、状态文案等主内容区域时：
  - 若当前未进入详情，则切换到该习惯详情
  - 若当前已在列表视图，则进入对应习惯详情
- 主区域不再承担“打开文档”职责

### 右侧辅助图标

- 现有日历图标替换为文档图标
- 点击后直接打开习惯定义所在文档，并定位到习惯定义 block
- 图标语义以“打开文档”为准，不再复用“详情入口”语义

### 详情入口的一致性

- 详情视图仍然保持现有返回逻辑
- 详情页右上角“打开文档”按钮保留不变
- 列表进入详情与其他详情入口共享同一套选择习惯逻辑，不新增单独状态

## 与 TodoDock 的关系

本调整会让 HabitDock 与 TodoDock 的主点击行为不同：

- `TodoDock`：主点击打开文档
- `HabitDock`：主点击进入详情

这是允许的刻意差异，因为两个面板的主任务不同：

- Todo 以“事项来源文档”为中心
- Habit 以“习惯状态与打卡详情”为中心

不为了机械一致性保留当前交互。

## 不做的事

- 不调整移动端 Habit 面板交互
- 不调整打卡按钮样式与位置
- 不改详情页结构
- 不修改 TodoDock 交互以追求一致

## 验收标准

1. 桌面端 HabitDock 列表项主区域点击后进入习惯详情，而不是打开文档
2. 列表项右侧辅助图标改为文档图标
3. 点击右侧文档图标后，打开对应习惯定义所在文档
4. 打卡按钮行为不变
5. 详情页返回列表逻辑不变
