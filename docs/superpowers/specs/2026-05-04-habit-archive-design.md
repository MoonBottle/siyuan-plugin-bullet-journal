# 习惯归档功能设计

## 背景

当前习惯系统只有“自然结束”这一种停止活跃的语义：

- `durationDays` 到期后，通过 `isEnded` 推导为已结束
- 已结束习惯仍然保留历史记录与统计

但这不足以覆盖用户主动停用习惯的场景，例如：

- 一个长期习惯不再继续，但并未设置 `坚持N天`
- 一个自然未结束的习惯希望退出主列表，停止提醒与打卡
- 一个已经形成历史数据的习惯，希望保留记录但不再参与“当前习惯”工作流

因此需要补充“归档”能力，作为与自然结束并列的第二种非活跃状态。

## 目标

为习惯增加用户主动控制的归档能力，满足以下目标：

- 归档状态可持久化，并与现有 Markdown 驱动模型一致
- 归档后习惯退出默认活跃列表
- 归档后停止提醒与今日打卡入口
- 历史记录、详情、统计仍可查看
- 归档与自然结束分别表达，不互相覆盖

## 非目标

- 不把归档实现为仅运行时状态
- 不把归档状态只写入 block attrs
- 不在本次设计中引入批量归档
- 不改动任务、番茄钟、待办的归档模型
- 不在本次设计中定义“已归档列表”的完整 UI 筛选方案，只预留能力

## 决策

采用 **Markdown 显式归档标记** 作为唯一归档持久化方式。

### 标记语法

在习惯定义行末尾增加归档标记：

```markdown
喝水 🎯2026-04-01 8杯 🔄每天 📦2026-05-04
```

其中：

- `📦` 表示该习惯已归档
- 后接归档日期 `YYYY-MM-DD`
- 标记只允许出现在习惯定义行，不允许出现在 record 行

### 为什么不用 block attr

因为当前习惯系统的核心原则是“文档即数据”，习惯定义、频率、提醒、打卡记录都直接写在 Markdown 中。若归档状态单独存到 block attr，会出现：

- 文档文本与真实状态分离
- 手写迁移、导入导出时语义不完整
- 调试与排查成本更高

因此不采用 attr-only 方案。

## 数据模型

在 `Habit` 上增加归档字段：

```typescript
interface Habit {
  // ...existing fields
  archivedAt?: string // YYYY-MM-DD，存在即表示已归档
}
```

归档状态推导规则：

```typescript
const isArchived = Boolean(habit.archivedAt)
```

### 与 `isEnded` 的关系

两者不是同一个维度：

- `isEnded`: 自然结束，由 `durationDays` / `endDate` 推导
- `isArchived`: 用户主动停用，由 `archivedAt` 决定

允许同时存在：

- 未结束但已归档
- 已结束但未归档
- 已结束且已归档

### 语义优先级

在“是否活跃”判断上：

```typescript
isActive = !isArchived && !isEnded
```

但在展示层保留两种状态来源：

- 已归档：用户主动停用
- 已结束：自然到期

不把二者合并成同一个文案。

## 解析规则

### 习惯定义行

新增解析结果：

| Markdown | 解析结果 |
|----------|----------|
| `喝水 🎯2026-04-01 8杯 🔄每天` | `Habit { archivedAt: undefined }` |
| `喝水 🎯2026-04-01 8杯 🔄每天 📦2026-05-04` | `Habit { archivedAt: '2026-05-04' }` |

### record 行

record 行不支持归档标记：

```markdown
喝水 3/8杯 📅2026-05-01
```

若 record 行出现 `📦`，按无效习惯 record 处理，不扩展兼容逻辑。

## 交互设计

### 归档入口

归档入口放在习惯详情视图右上角操作区，和“打开文档”同层级。

桌面端：

- 未归档：显示“归档”动作
- 已归档：显示“取消归档”动作

移动端：

- 在习惯详情 bottom sheet / panel 的头部操作区提供同样能力

不在列表行直接放归档按钮，避免与“打卡”“打开文档”“进入详情”竞争主操作位。

### 归档确认

执行归档时弹确认：

- 标题：归档习惯
- 说明：归档后将停止提醒，并从默认习惯列表中隐藏；历史记录会保留

取消归档不弹确认，直接恢复。

### 列表展示

默认活跃列表仅展示未归档习惯。

本次先定义行为，不强制要求立即提供“查看已归档习惯”入口。实现上需要保留后续扩展能力，例如：

- 设置开关
- 独立筛选
- 独立分组

### 详情展示

若通过导航或未来的已归档视图进入已归档习惯详情：

- 详情仍可打开
- 月历、统计、日志仍可查看
- 打卡入口禁用
- 显示“已归档”状态提示

## 行为规则

### 提醒

已归档习惯不再参与提醒调度：

```typescript
if (habit.archivedAt) skip
```

这条规则优先于频率、时间、日期是否命中。

### 打卡

已归档习惯不能再作为“当前活跃习惯”打卡：

- Dock 列表中默认不可见，因此不会出现打卡按钮
- 若从详情或历史入口进入已归档习惯：
  - 二元型打卡按钮禁用
  - 计数型 +1 按钮禁用
  - `/dk` 命中习惯定义行时提示“习惯已归档”

### 历史记录

归档不修改任何既有 record：

- 不删除
- 不迁移
- 不重写
- 不补写归档说明到 record

### 统计

历史统计保留，但默认只服务于当前正在查看的习惯详情。

对全局/列表层的影响：

- 默认活跃列表统计不混入已归档习惯
- 单个已归档习惯详情仍可完整查看累计数据、月数据、连续数据

### 打开文档

归档不影响文档跳转：

- 仍然可以打开习惯定义所在 block
- 文档是归档习惯的最终可审计来源

## 编辑规则

### 创建时

新建习惯不允许直接创建为已归档。

理由：

- 创建即归档没有明确业务价值
- 会增加创建表单复杂度
- 用户心智应为“先创建，再归档”

### 编辑时

归档状态不通过现有“创建/编辑习惯弹框”管理，避免把结构化编辑和生命周期操作混在一起。

归档/取消归档应走独立操作按钮。

### 文本更新方式

归档动作本质是对习惯定义行做一次文本更新：

- 未归档 -> 追加 `📦YYYY-MM-DD`
- 已归档 -> 移除 `📦YYYY-MM-DD`

不改动该行其他字段顺序与内容。

## 斜杠命令规则

### `/dk`

补充一条规则：

| 触发行 | 当前状态 | 期望行为 |
|--------|----------|----------|
| 习惯定义行 | 已归档 | 提示「习惯已归档」 |

### `/xg`

编辑习惯时不在弹框里提供归档切换，保持编辑职责单一。

## 错误处理

### 归档标记格式错误

若用户手写了错误格式：

```markdown
喝水 🎯2026-04-01 8杯 🔄每天 📦今天
```

处理规则：

- 不识别为已归档
- 不报错中断整条习惯解析
- 保持其余习惯字段正常解析

### 归档动作写回失败

若更新 block 失败：

- 保持当前 UI 状态不切换
- 提示用户归档失败
- 不做本地假归档

## 测试重点

### 解析

- 能解析 `📦YYYY-MM-DD`
- 错误格式不生效
- record 行不接受归档标记

### 领域规则

- `isArchived` 与 `isEnded` 可独立存在
- 活跃性判断为 `!isArchived && !isEnded`

### UI

- 详情页可归档/取消归档
- 默认列表隐藏已归档习惯
- 已归档详情打卡按钮禁用

### 服务

- 已归档习惯不参与提醒
- `/dk` 命中已归档习惯时返回归档提示

## 验收标准

1. 用户可以在习惯详情中归档或取消归档习惯
2. 归档状态以 Markdown 形式持久化在习惯定义行中
3. 已归档习惯默认不出现在活跃习惯列表
4. 已归档习惯不再触发提醒
5. 已归档习惯不能继续打卡
6. 历史记录、统计、详情查看能力保留
7. `isArchived` 与 `isEnded` 是两个独立语义，不互相覆盖
