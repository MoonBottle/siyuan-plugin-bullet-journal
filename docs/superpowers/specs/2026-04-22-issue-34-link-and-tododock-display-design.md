# Issue #34 链接解析与 Todo Dock 展示设计

## 一、功能概述

本设计解决 issue #34 的两个诉求：

1. **补充块引用链接解析**：支持将任务下方、事项下方“单独一行的思源块引用”识别为链接
2. **增强 Todo Dock 信息密度**：支持在事项卡片上直接展示链接、提醒、重复信息，并通过右上角菜单控制显示

### 1.1 设计目标

- **遵循现有解析边界**：沿用 `core.ts` 中现有的任务区 / 事项区分区逻辑，不重写 MarkdownParser 职责
- **轻量扩展 Link 模型**：为链接增加类型信息，支持展示层按类型区分样式
- **默认保守**：Todo Dock 默认不展示新增信息，通过用户偏好开关开启
- **主题一致性**：链接类型的视觉区分使用思源主题变量，不引入硬编码色值
- **紧凑展示**：Dock 中任务链接和事项链接合并展示，减少层级感

## 二、需求范围

### 2.1 已有能力

当前系统已经支持：

- 项目名、任务名、事项内容中的**行内块引用**解析
- 任务下方、事项下方的普通 Markdown 链接 `[名称](URL)` 解析
- 事项详情弹框中查看 task/item/project 层级链接

### 2.2 本次新增能力

本次新增：

- 任务下方、事项下方“**单独一行块引用**”解析为 links
- Todo Dock 菜单增加两个持久化开关：
  - 显示链接
  - 显示提醒与重复
- Todo Dock 事项卡片中合并展示当前事项可访问的链接
- Dock 与详情中的链接按类型增加不同背景色
- 用户文档补充块引用链接格式与展示规则

### 2.3 非目标

本次不做：

- 不重构 `MarkdownParser` 的解析流程
- 不修改“行内块引用”的现有语义
- 不改变详情弹框中 project/task/item 分层展示方式
- 不新增设置页入口，仅通过 Todo Dock 菜单管理这两个显示开关

## 三、解析规则设计

### 3.1 区域归属规则

解析归属遵循 `src/parser/core.ts` 现有状态机：

- **任务下方的单独一行块引用** → 归属 `currentTask.links`
- **事项下方的单独一行块引用** → 归属 `currentItem.links`

不改变现有“谁属于任务区、谁属于事项区”的判断逻辑，只是在相应分支内新增块引用链接处理。

### 3.2 支持的块引用形式

支持思源 Kramdown 中的单独一行块引用：

```markdown
((20260422075729-q6vs0km 'ddddd的的'))
```

以及双引号形式：

```markdown
((20260422075729-q6vs0km "ddddd的的"))
```

### 3.3 单独一行块引用的判定

仅当某一行在去除列表前缀、块属性后，内容整体匹配块引用格式时，才将其视为“下方链接块”。

这样可以避免：

- 将正文中混杂的普通文本误识别为附属链接
- 与现有 `parseItemLine`、`parseTaskLine` 中的行内块引用处理相互干扰

### 3.4 行内块引用保持不变

以下场景继续沿用现有行为：

```markdown
首页((20260310210016-gkixdit '测试'))改版 #任务
```

```markdown
确定设计风格((20260310210016-gkixdit "别名")) @2026-03-09
```

解析结果保持：

- 显示文本中用锚文本替换块引用
- 同时向对应 task/item 的 `links` 追加一条块引用链接

### 3.5 多日期事项

多日期事项的下方块引用链接与现有普通 links 行为保持一致：

- 链接写在事项下方
- 展开后的每个 item 都携带相同的 link 集合

## 四、数据模型设计

### 4.1 Link 类型扩展

在 `src/types/models.ts` 中扩展 `Link`：

```ts
export type LinkType = 'external' | 'siyuan' | 'block-ref';

export interface Link {
  name: string;
  url: string;
  type?: LinkType;
}
```

`type` 先保持可选，兼容现有代码路径；所有新增和更新后的解析结果都应尽量写入 `type`。

### 4.2 类型判定规则

| 类型 | 触发方式 | 示例 |
|------|---------|------|
| `external` | 普通外部 URL | `https://example.com` |
| `siyuan` | 显式思源协议链接 | `siyuan://blocks/20260220112000` |
| `block-ref` | 由 `((blockId '锚文本'))` 解析得到 | `((202604... '设计稿'))` |

### 4.3 块引用链接的数据结构

单独一行块引用与行内块引用都统一落成：

```ts
{
  name: '设计稿',
  url: 'siyuan://blocks/20260422075729-q6vs0km',
  type: 'block-ref',
}
```

无锚文本时沿用现有兜底命名逻辑。

## 五、Todo Dock 交互设计

### 5.1 菜单开关

在 `src/tabs/DesktopTodoDock.vue` 的右上角更多菜单中新增两个持久化选项：

- 显示链接 / 隐藏链接
- 显示提醒与重复 / 隐藏提醒与重复

### 5.2 持久化规则

两个开关均写入 `settingsStore.todoDock`，默认值为 `false`：

```ts
todoDock: {
  hideCompleted: boolean,
  hideAbandoned: boolean,
  showLinks: boolean,
  showReminderAndRecurring: boolean,
}
```

切换后应立即：

- 更新本地 store
- 持久化到插件设置
- 触发 Dock 响应式刷新

### 5.3 卡片展示原则

开启开关后，在 `TodoSidebar.vue` 的事项卡片正文下方增加紧凑信息区。

展示规则：

- `showLinks = true` 时显示链接区
- `showReminderAndRecurring = true` 时显示提醒 / 重复区
- 无数据时不占位
- 不移除现有详情按钮

### 5.4 Dock 中链接合并展示

虽然数据仍分属 `task.links` 与 `item.links`，但在 Todo Dock 展示层统一合并为一个紧凑链接区：

- 顺序：`item.links` 在前，`task.links` 在后
- 去重键建议为：`name + url + type`
- 不显示“任务链接”“事项链接”标题

这样可以：

- 降低视觉层级
- 提高信息密度
- 与“快速查看当前事项相关链接”的使用场景更匹配

### 5.5 提醒 / 重复展示与交互

提醒与重复在 Todo Dock 中不能只是静态 tag，而要参考 `ItemDetailDialog.vue` 的现有 UI 和交互语义，作为可点击操作入口展示：

- 提醒按钮示例：`⏰ 提前10分钟`
- 绝对提醒示例：`⏰ 08:00`
- 重复按钮示例：`🔁 每周一三五`
- 带结束条件时，Dock 中保持紧凑展示，不展开成长说明

交互要求：

- 点击提醒按钮时，直接打开提醒设置弹框
- 点击重复按钮时，直接打开重复设置弹框
- tooltip、只读态、已激活态的交互语义尽量与 `ItemDetailDialog.vue` 保持一致
- 多日期事项仍遵循现有规则，不开放重复设置

目的：

- 在不打开详情弹框的前提下，让用户既能看到信息，也能直接操作
- 保持卡片高度可控，同时复用已有交互认知

## 六、视觉设计

### 6.1 链接类型区分

Dock 与详情中展示的链接都按 `Link.type` 区分背景色。

### 6.2 颜色来源

颜色实现必须使用思源主题变量组合，不直接写死十六进制颜色。

推荐复用的变量来源包括但不限于：

- `--b3-theme-primary`
- `--b3-theme-primary-light`
- `--b3-theme-surface`
- `--b3-theme-surface-lighter`
- `--b3-border-color`
- `--b3-theme-on-surface`

### 6.3 视觉语义

建议视觉语义如下：

- `external`：中性链接标签风格
- `siyuan`：主色弱强调，表示思源内部跳转
- `block-ref`：与 `siyuan` 可区分的另一组弱强调背景，表示“引用而非显式链接”

要求：

- 三类链接都要保持低饱和、轻量，不压过事项正文
- 同一套 class 规则尽量复用于 Dock 与详情弹框，避免样式漂移

## 十、风险与约束

### 10.1 解析风险

如果“单独一行块引用”的判定过宽，可能会把非链接内容误挂到 task/item links。为避免该问题，本次只接受“整行为块引用”的附属块判定。

### 10.2 UI 风险

当事项拥有较多链接时，卡片高度可能增长。通过以下方式控制：

- 使用紧凑 tag 样式
- 支持自然换行
- 不增加二级标题

### 10.3 兼容性

`Link.type` 采用可选字段，可减少对旧逻辑的冲击；展示层应为无 `type` 的旧数据提供安全降级样式。

---

## 十一、结论

本设计采用最小侵入方案，在不改变现有解析架构的前提下：

- 补齐任务/事项下方单独一行块引用的链接解析
- 为链接增加类型信息，支持 Dock 与详情的视觉区分
- 在 Todo Dock 中通过持久化开关，将链接、提醒、重复前移到事项卡片上
- 使用思源主题变量实现类型化链接背景，保持主题一致性

该方案满足 issue #34 的核心需求，改动边界清晰，适合进入实现计划阶段。
