# 四象限悬浮聚焦编辑设计

## 一、背景

四象限 Tab 已经提供了按优先级映射的任务分象限视图，适合做快速筛选、优先级调整和批量浏览。但当前用户要编辑某个事项及其子块时，仍需要点击跳转到原文档，这会打断在四象限中的连续处理流程。

思源原生引用块悬浮并不是简单 tooltip，而是一套可编辑的块级浮层能力。源码中：

- `app/src/block/popover.ts` 负责识别触发目标并创建浮层
- `app/src/block/Panel.ts` 负责创建浮层容器并挂载 `Protyle`
- `BlockPanel` 内部会根据目标块是否为根文档，选择 `CB_GET_ALL` 或 `CB_GET_CONTEXT`

同时，`node_modules/siyuan` 已导出 `Protyle`、`BlockPanel`、`TProtyleAction` 等类型与类，说明插件侧可以直接挂载思源原生编辑实例，而不是自己实现一套伪编辑器。

本次设计目标是在四象限卡片 hover 时提供类似“聚焦到任务块及所有子块”的悬浮编辑体验，但实现边界由插件自行控制，以便后续复用于其他视图。

---

## 二、目标

### 2.1 目标行为

在桌面端四象限 Tab 中，鼠标悬浮到事项卡片后，延迟打开一个悬浮聚焦编辑浮层：

- 展示目标任务块及其所有子块
- 浮层内可直接编辑，使用思源原生编辑能力
- 鼠标可从卡片移入浮层而不闪退
- 离开卡片与浮层后延迟关闭
- 当前卡片点击跳转文档的主行为仍然保留

### 2.2 复用目标

首版仅在 `QuadrantTab` 启用，但实现上应抽成通用能力，后续可供 Todo Dock、Calendar、Gantt 或其他块列表视图接入。

### 2.3 硬约束

本功能的编辑区必须使用思源原生 `Protyle` 实例承载，不自己实现任何编辑能力，包括但不限于：

- 不自己拼装可编辑块 DOM
- 不自己模拟块级菜单/快捷键/输入逻辑
- 不自己实现“子块编辑区”

插件只负责：

- hover 触发时机
- 浮层容器与定位
- 生命周期与显隐逻辑
- 将 `blockId` 和挂载节点交给原生 `Protyle`

---

## 三、非目标

本次不做：

- 不在 Todo Dock / Calendar / Gantt 中同时开启该交互
- 不做移动端替代交互
- 不实现多个悬浮编辑浮层并行管理
- 不复刻思源原生 `BlockPanel` 的完整能力，如 pin、拖拽移动、新窗口打开、层级链式弹层
- 不引入自定义编辑器或只读预览渲染器替代 `Protyle`

---

## 四、方案选择

### 4.1 被采纳方案

采用“通用悬浮聚焦能力层 + 四象限先接入”的方案：

1. 抽出通用状态与交互层
2. 抽出通用浮层组件
3. 在浮层内部直接挂载思源原生 `Protyle`
4. 四象限卡片只提供 `blockId` 与锚点元素

### 4.2 不直接复用 `BlockPanel` 的原因

虽然 `node_modules/siyuan` 已导出 `BlockPanel`，但它依赖较多思源前端内部约定和全局状态，例如：

- `window.siyuan.blockPanels`
- 全局 z-index / menu 层级处理
- 原生触发目标识别
- pin / close / open 等整套浮层工具栏语义

如果插件直接绑定整套 `BlockPanel` 生命周期，后续维护与兼容性风险更高。更合适的做法是：

- 内容区复用原生 `Protyle`
- 数据范围策略参考 `BlockPanel`
- 浮层壳子与触发逻辑由插件自己掌控

这能满足“原生编辑”要求，同时把依赖边界控制在更小范围内。

---

## 五、架构设计

### 5.1 能力分层

新增两层通用能力：

1. `useBlockFocusPreview()`
2. `BlockFocusPreviewPopover.vue`

#### `useBlockFocusPreview()` 职责

- 管理当前激活的 `blockId`
- 管理当前锚点元素
- 管理显示、隐藏、延迟定时器
- 管理加载中与错误状态
- 处理“卡片 -> 浮层”的 hover 保活逻辑
- 对外暴露统一的接入接口

#### `BlockFocusPreviewPopover.vue` 职责

- 渲染悬浮容器
- 负责定位
- 提供原生编辑挂载点
- 在生命周期中创建、销毁 `Protyle`
- 处理浮层内部 hover / close / error / loading 展示

### 5.2 与业务视图的关系

`QuadrantTab` 不直接持有编辑实现细节，只负责：

- 在卡片 hover 时传入 `blockId`
- 将卡片 DOM 作为锚点传给能力层
- 在拖拽等冲突场景下决定是否抑制预览

后续其他视图如果要复用，应只需要复用同一套接口，而不需要复制悬浮编辑内部逻辑。

---

## 六、原生编辑实现策略

### 6.1 使用 `Protyle` 作为唯一编辑核心

浮层内容区必须直接创建思源原生 `Protyle`：

- 来源：`node_modules/siyuan` 导出的 `Protyle`
- 挂载方式：`new Protyle(app, mountElement, options)`

这与思源源码中搜索预览、历史对比、反链面板、卡片预览、块悬浮等场景保持一致。

### 6.2 内容范围策略

聚焦内容范围直接参考思源原生 `BlockPanel`：

- 若目标 `blockId` 不是根文档块：使用 `CB_GET_ALL`
- 若目标 `blockId` 是根文档块：使用 `CB_GET_CONTEXT`

也就是：

- 任务块场景下，优先保证“目标块 + 所有子块”
- 文档根块场景下，保留文档上下文打开语义

### 6.3 建议依赖的原生参数

当前设计明确依赖的 `IProtyleOptions` 字段包括：

- `blockId`
- `action`
- `render`
- `typewriterMode`
- `after`

首版不依赖 `backlinkData`、`history` 等特殊场景参数。

### 6.4 渲染配置目标

浮层内 `Protyle` 应尽量保留原生编辑可用性，同时避免视觉噪音过高。建议配置方向：

- `gutter: true`
- `scroll: true`
- `breadcrumbDocName: true`
- `title`: 仅在根文档场景显示
- `typewriterMode: false`

是否显示完整 breadcrumb、背景、顶部标题栏，可在实现阶段根据四象限容器空间再做细调，但不能削弱原生可编辑性。

## 七、交互设计

### 7.1 打开时机

首版使用 hover 打开：

- 鼠标移入卡片后延迟显示
- 避免光标快速掠过时频繁创建 `Protyle`

### 7.2 关闭时机

- 鼠标离开卡片后，不立即关闭
- 若鼠标进入浮层，则保持打开
- 仅当鼠标离开卡片与浮层后，再延迟关闭

### 7.3 与现有点击行为共存

卡片当前点击跳转文档行为保持不变。hover 只是新增一条“不中断工作流的编辑入口”，不是替换当前点击语义。

### 7.4 与拖拽交互的冲突处理

四象限已经支持跨象限拖拽改优先级，因此需要显式避免冲突：

- 拖拽开始时，禁止打开新的 hover 预览
- 正在拖拽时，如果已有浮层，可按实现复杂度选择保持或关闭，但不能误触发新实例
- 浮层不作为拖拽目标语义的一部分

### 7.5 单实例策略

首版仅维护一个激活中的悬浮编辑实例：

- 新卡片触发时，关闭旧实例并切换到新 `blockId`
- 不支持多浮层并行

这可以显著降低焦点管理、z-index 和销毁顺序复杂度。

---

## 八、接口设计

### 8.1 通用接入接口

对业务视图暴露的接口应尽量小，只围绕：

- `blockId`
- `anchorElement`
- `enabled`

接入形式可以是 composable 暴露的方法，也可以配合轻量指令/事件绑定层，但能力层不应耦合四象限特有语义，如：

- priority
- quadrant key
- drag state 之外的业务筛选状态

### 8.2 四象限接入方式

四象限首版只在事项卡片层接入：

- 卡片 hover 触发预览
- 卡片 `blockId` 为空时不启用
- 列表仍由 `TodoSidebar` 渲染

如需给 `TodoSidebar` 增加最小扩展，应仅用于向外暴露 hover 触发上下文，不把预览实现写死在 `TodoSidebar` 内部。

---

## 九、刷新与数据一致性

### 9.1 编辑后的数据流

浮层内编辑完成后，仍应走思源和插件现有刷新链路：

- 优先依赖思源原生编辑写回
- 再通过现有 `eventBus` / `BroadcastChannel` / store 刷新机制更新四象限列表

不单独为悬浮编辑设计新的本地同步协议。

### 9.2 一致性目标

用户在悬浮编辑中修改任务块内容、子块、日期或状态后：

- 思源文档内容为真实源
- 四象限卡片在后续刷新中反映最新结果
- 不要求通过局部手改数组实现“即时假同步”

---

## 十、测试与验证

### 10.1 单测重点

- hover 延迟显示与延迟关闭
- 卡片移入浮层时不闪退
- 拖拽开始时抑制预览打开
- 单实例切换时旧实例正确销毁
- `blockId` 缺失时不创建预览

### 10.2 集成验证重点

- 任务块场景下，浮层能加载目标块及所有子块
- 浮层内确实使用原生 `Protyle`，而非自定义编辑 DOM
- 浮层内可直接编辑主块与子块
- 编辑后四象限卡片经现有刷新链路更新
- 卡片点击跳转文档行为不受影响

### 10.3 人工验证重点

- 长任务、多子块任务在浮层中的高度和滚动表现
- 不同象限、不同筛选条件下预览目标是否正确
- 与右键菜单、详情弹窗、拖拽改优先级之间是否有明显焦点冲突

---

## 十一、实现边界总结

这次能力的实现原则是：

- **编辑能力原生化**：只使用思源原生 `Protyle`
- **触发层插件化**：hover、定位、显隐由插件接管
- **首版范围克制**：只在四象限启用，单实例，桌面端
- **架构预留复用**：命名与接口不绑定四象限专用语义

这使得功能既能满足“在四象限里连续处理任务，不跳文档打断工作流”的目标，也不会把插件绑定到整套思源内部浮层生命周期上。
