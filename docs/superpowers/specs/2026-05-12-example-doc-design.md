# 示例文档与数据格式文档重构设计

## 背景

当前示例文档和 `docs/user-guide/data-format.md` 的问题主要有三类：

1. 示例文档更像“功能句子列表”，不像用户可以直接模仿的真实笔记。
2. `data-format.md` 同时承担“新手入门”和“完整语法手册”两类职责，信息密度过高。
3. 任务、事项、习惯、番茄钟、斜杠命令等能力虽然覆盖较全，但缺少符合真实使用频率的教学顺序。

本次设计目标不是减少功能覆盖，而是重排内容组织，让新用户更快上手，同时保留完整参考价值。

## 目标

重构后需要同时满足以下目标：

- 对新用户友好，不要求先理解完整模型才能开始使用。
- 采用分层展示，前半段可快速上手，后半段再展开能力。
- 有教学意义，能解释“什么时候这样写”而不是只列语法。
- 相对全面，能够让用户意识到插件不止支持基础待办。

## 核心判断

用户的真实心智路径应为：

1. 先记一条待办
2. 再开始执行这条待办
3. 待办变多后，再按任务组织
4. 需要时再加提醒、重复、优先级
5. 最后再了解习惯、多日期、链接等扩展能力

因此，示例内容的优先顺序应为：

1. 独立事项
2. 事项 + 番茄钟
3. 任务 + 事项
4. 优先级 / 提醒 / 重复
5. 习惯 / 多日期 / 链接

斜杠命令属于“降低输入门槛”的高频能力，不应被放到附录区域，而应前置到主教学路径中。

## 信息架构

### 1. 示例文档的职责

示例文档应扮演“真实样板”的角色：

- 用户打开后可以直接照着写
- 优先展示推荐写法
- 让用户看到插件的典型使用路径
- 不负责穷举所有等价语法和边界规则

### 2. `data-format.md` 的职责

`data-format.md` 应拆成两层职责：

- 上半部分：入门路径，帮助用户理解如何开始使用
- 下半部分：完整参考，提供完整格式、别名、规则细节与 FAQ

它不应再以“项目区 / 任务区 / 事项区”的纯语法分类作为开头，而应先按用户使用路径组织。

## 示例文档设计

### 第一层：快速开始（独立事项）

目的：

- 告诉用户不必先创建任务
- 适配 daily note / 随手待办 的高频场景

内容要求：

- 3 到 5 条独立事项
- 包含今天、明天、带具体时间的事项
- 用极短提示说明可用 `/today`、`/tomorrow`

这一层只解决“我第一条怎么写”。

### 第二层：事项 + 番茄钟

目的：

- 告诉用户事项不是静态记录，而可以直接进入执行
- 建立事项与番茄钟记录的关联心智

内容要求：

- 1 条事项
- 1 条挂在该事项下的番茄记录
- 1 句简短说明：专注结束后会自动追加，也支持手写

这一层只解决“我记录后怎么开始做”。

### 第三层：任务 + 事项

目的：

- 在用户已经理解事项后，再引入任务作为组织结构
- 强调任务是目标/分组，事项是带日期的行动

内容要求：

- 1 个真实感强的项目
- 2 个任务
- 每个任务下 2 到 3 个事项
- 自然包含完成态，而不是单独写教学式句子

辅助提示：

- 一行转任务可用 `/task`
- 完成事项可用 `/done`

### 第四层：增强事项

目的：

- 展示事项在执行层面的增强能力

内容要求：

- 优先级 1 例
- 提醒 1 例
- 重复 1 例

表达原则：

- 每个能力只放一个最典型场景
- 强调“什么时候用”，不在示例文档里讲全量变体

配套斜杠命令提示：

- `/priority`
- `/reminder`
- `/recurring`

### 第五层：更多玩法

目的：

- 告知用户插件还有更丰富的能力
- 但不打断前半段的入门主线

内容要求：

- 习惯 1 例
- 多日期 1 例
- 链接 1 例

这里仅做露出，不展开复杂规则。

### 第六层：极简斜杠命令速查

目的：

- 让用户在看完样例后快速记住最常用输入捷径

建议只保留 6 到 8 个命令：

- `/today`
- `/tomorrow`
- `/task`
- `/done`
- `/detail`
- `/reminder`
- `/recurring`
- `/focus`（或当前实际专注入口命令）

这里必须是“速查表”，不是完整命令大全。

## `data-format.md` 结构设计

### 上半部分：怎么开始

建议章节顺序：

1. 先写一条独立事项
2. 事项如何配合番茄钟
3. 待办多了，再使用任务组织
4. 给事项增加优先级、提醒、重复
5. 还有习惯、多日期、链接等能力
6. 常用斜杠命令速查

每节统一采用：

- 一句“什么时候用”
- 一个最短示例
- 一个轻提示（必要时附斜杠命令）

### 下半部分：完整格式参考

保留并整理现有信息：

- 所有标记说明
- Emoji / 文本等价写法
- 状态标签
- 提醒规则
- 重复规则
- 多日期格式
- 习惯格式
- 番茄钟格式
- 链接解析规则
- FAQ

但这些内容应明确降级为“参考手册”，不作为首次阅读主线。

## 示例文档内容取舍

### 主示例中应保留

- 独立事项
- 事项与番茄钟的关联
- 任务 + 事项层级
- 已完成状态
- 优先级 1 例
- 提醒 1 例
- 重复 1 例
- 习惯 1 例
- 极简斜杠命令提示

### 主示例中不应重点展示

- 中英文双语等价写法并排
- 同一能力的多种语法变体
- 秒级时间
- 复杂多日期混合范围
- 复杂重复规则
- 链接解析的所有细节
- 大而全的斜杠命令清单
- 低频状态在前半段过度占位

这些内容可保留在参考部分，但不应进入新手主线。

## 实施边界

本次设计只涉及内容组织与教学路径，不涉及解析器能力变更。

预期后续实施文件：

- `src/utils/exampleDocUtils.ts`
- `docs/user-guide/data-format.md`

不要求同步改动 parser、store 或 UI 逻辑。

## 验收标准

完成后应满足以下标准：

1. 新用户打开示例文档后，能在 1 分钟内抄出第一条可用事项。
2. 用户能自然理解“独立事项可直接用于 daily note”。
3. 用户能在前半段看到事项与番茄钟的关联，而不是把番茄钟理解成独立功能。
4. 用户能在阅读主线后理解“任务是组织结构，事项是执行单元”。
5. 文档前半部分不会出现大段等价语法或完整规则表。
6. 完整能力仍能在后半部分查到，不丢失参考价值。

## 建议实施顺序

1. 先重写 `exampleDocUtils.ts` 的示例内容
2. 再重排 `data-format.md` 的章节结构
3. 最后检查示例文档与用户指南中的术语是否一致
