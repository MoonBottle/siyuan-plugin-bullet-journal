# 工作台习惯打卡 Widget 设计

## 一、背景

当前工作台里已经有两种与习惯相关的承载方式：

1. `habit` 工作台视图
   - 已经实现为左右两栏
   - 左侧是周摘要 + 习惯列表
   - 右侧是习惯详情

2. `habitWeek` 仪表盘 widget
   - 当前只渲染 `HabitWeekBar`
   - 只能看到一周摘要，不能直接浏览习惯列表，也不能进入详情

这导致工作台中的习惯能力出现明显割裂：

- 如果用户只在仪表盘工作，只能看到周摘要，无法直接处理习惯
- 如果要看详情或打卡，需要切到完整 `habit` 视图
- 与 todo / calendar widget 正在增强为“可在仪表盘内直接处理”的方向不一致

本次目标是在不破坏现有侧栏习惯交互的前提下，把 `habitWeek` widget 升级为一个真正可用的习惯工作入口。

---

## 二、目标

### 2.1 目标行为

将工作台中的 `habitWeek` widget 升级为：

- 顶部展示一周摘要条
- 下方展示习惯列表
- 点击习惯项后，以弹框形式展示完整习惯详情
- 详情内容与交互尽量复用现有习惯详情实现
- 支持分组筛选配置，并持久化到工作台 JSON

### 2.2 用户体验目标

用户在仪表盘里应该可以：

- 快速扫一眼本周习惯完成情况
- 直接找到某个习惯
- 直接完成打卡 / 增量打卡
- 在不跳离工作台、不切换到独立视图的情况下查看该习惯的月历、统计、记录

### 2.3 非目标

本次不做：

- 不改变桌面侧栏 `HabitDock` 的交互模型
- 不改变移动端习惯面板
- 不新增复杂筛选条件，首版只支持分组筛选
- 不把习惯详情重新实现成一套 widget 专属 UI
- 不让 widget 内直接展开右侧详情区域

---

## 三、方案选择

本次采用方案 A：

> 抽出“习惯工作区核心组件”，由侧栏和 widget 共享核心内容；侧栏继续使用左右栏承载详情，widget 使用弹框承载详情。

### 3.1 选中 A 的原因

- 已有 `DesktopHabitDock` 和 `WorkbenchHabitView` 都具备可复用的结构基础
- 用户明确要求“详情内容与交互尽量完整复用”
- 工作台 widget 后续还可能继续增强，先把复用边界理顺更稳

### 3.2 不采用 B / C 的原因

- 单独做一套 widget 版习惯列表，会形成两套习惯展示逻辑，后续很容易漂移
- 只复用详情、不复用列表和状态计算，会导致分组筛选、选中态、打卡刷新重复实现

---

## 四、设计原则

### 4.1 业务逻辑只保留一套

以下逻辑必须尽量统一来源：

- 习惯列表数据
- 周摘要数据
- 当日状态 / 周期状态
- 统计计算
- 打卡后的刷新回流
- 当前选中习惯的详情计算

### 4.2 容器与内容分离

“展示什么内容”与“详情承载在什么容器里”必须分开：

- 侧栏模式：详情承载在右栏
- widget 模式：详情承载在弹框

这样后续如果别的地方也需要嵌入习惯工作区，可以复用内容层，不被固定布局绑死。

### 4.3 不影响现有侧栏体验

`DesktopHabitDock` 仍然保持：

- 列表点击后切换详情视图
- 顶栏返回列表
- 原有刷新与打开文档行为

本次改造不能要求用户重新适应侧栏习惯交互。

---

## 五、信息架构

### 5.1 Widget 结构

升级后的 `habitWeek` widget 分为两部分：

1. 顶部摘要区
   - 使用现有 `HabitWeekBar`
   - 作为本周习惯完成情况概览

2. 列表区
   - 展示当前筛选结果下的习惯列表
   - 列表项复用现有 `HabitListItem`
   - 列表项支持：
     - 打卡
     - 计数型增量打卡
     - 打开详情

### 5.2 详情呈现

在工作台 widget 中，点击某个习惯项后：

- 打开一个思源原生 `Dialog`
- Dialog 内承载完整习惯详情内容
- 详情内容尽量复用当前 `HabitMonthCalendar + HabitStatsCards + HabitRecordLog` 的组合
- 保留刷新、打开文档等上下文操作

### 5.3 与完整习惯视图的关系

工作台完整 `habit` 视图继续保留两栏模式：

- 左栏：周摘要 + 列表
- 右栏：详情

仪表盘 widget 则变成：

- 卡片内：周摘要 + 列表
- 点击后：弹框详情

因此，两者共享的是“核心内容层”，不共享“详情承载布局”。

---

## 六、组件拆分

### 6.1 目标拆分

建议抽出以下层次：

1. `HabitWorkspaceCore`
   - 负责习惯数据获取、分组过滤、选中项、状态计算、刷新
   - 输出摘要、列表、详情所需数据与操作

2. `HabitWorkspaceListPane`
   - 负责渲染 `HabitWeekBar + HabitListItem[]`
   - 只关心展示与交互事件，不持有业务源数据

3. `HabitWorkspaceDetailPane`
   - 负责渲染详情主体
   - 复用现有：
     - `HabitMonthCalendar`
     - `HabitStatsCards`
     - `HabitRecordLog`

4. 容器层
   - `DesktopHabitDock`：列表 / 详情切换容器
   - `WorkbenchHabitView`：左右两栏容器
   - `HabitWeekWidget`：列表 + 弹框详情容器

### 6.2 不要求一次性大重构

首轮改造不要求把所有习惯相关组件一次性抽到最完美。

允许保守演进，但必须满足两点：

- 不能复制整套详情实现
- 不能让 widget 和侧栏各自维护一套独立状态计算

如果某些逻辑暂时只能通过抽 composable 达成，也可以接受。

## 七、数据与配置

### 7.1 Widget 配置模型

`habitWeek` widget 新增配置：

```ts
type WorkbenchHabitWeekWidgetConfig = {
  groupId?: string;
};
```

首版只支持 `groupId`。

### 7.2 配置入口

widget 的“配置”弹框首版只提供分组选择：

- 全部分组
- 某个具体分组

配置方式与 `todoList` / `miniCalendar` widget 保持一致：

- 使用工作台 widget 配置弹框
- 保存到 workbench JSON
- 不写回全局 settings

### 7.3 数据过滤

widget 渲染时按配置过滤习惯：

- `groupId` 为空时展示全部习惯
- `groupId` 有值时只展示该分组下习惯

完整 `habit` 工作台视图与桌面侧栏保持当前行为，不被此 widget 配置影响。

---

## 八、交互设计

### 8.1 列表项交互

widget 中的习惯列表项应保留以下能力：

- 点击主区域：打开详情弹框
- 点击打卡按钮：直接打卡
- 点击计数型加一按钮：直接增加一次

列表区本身不展示详情展开态，不做嵌入式折叠。

### 8.2 详情弹框交互

弹框内保留完整详情体验：

- 月历视图
- 统计卡片
- 打卡日志
- 刷新按钮
- 打开文档按钮

弹框关闭后：

- 不需要把“已打开详情”状态保留在 widget 卡片里
- 下次再次点击某习惯时重新打开

### 8.3 打卡后的刷新回流

无论在以下哪个位置触发打卡：

- widget 列表项
- widget 详情弹框
- 其他视图广播刷新

都应保证以下数据同步：

- widget 顶部周摘要
- widget 列表状态
- 已打开的详情弹框内容

如果详情弹框对应的习惯仍存在，应优先同步更新，而不是强制关闭弹框。

### 8.4 空状态

当当前分组下没有任何习惯时：

- widget 顶部仍可保留空摘要条或安全降级为空占位
- 列表区域显示轻量空状态
- 不显示详情入口

---

## 九、弹框承载方案

### 9.1 承载方式

工作台习惯详情采用思源原生 `Dialog` 承载。

原因：

- 与现有工作台配置弹框风格一致
- 生命周期清晰
- 不需要在 widget 卡片内部额外管理浮层层级

### 9.2 生命周期要求

弹框打开时：

- 挂载详情组件
- 注入共享 Pinia
- 接收当前 habit、当前日期、当前视图月等必要上下文

弹框关闭时：

- 正常卸载 Vue 应用
- 清理事件订阅
- 清理临时选中状态

必须避免因弹框多次打开关闭导致内存泄漏或重复刷新监听。

---

## 十、实现风险

### 10.1 现有详情逻辑对容器的隐式依赖

如果现有习惯详情逻辑默认假设：

- 一定存在右栏
- 一定有“返回列表”按钮
- 一定运行在 dock 容器里

则需要把这些依赖下沉到容器层，不能继续留在详情内容层。

### 10.2 数据刷新的一致性

习惯视图当前涉及：

- `projectStore.refresh`
- `eventBus` 的 `DATA_REFRESH`
- `BroadcastChannel`

widget 弹框接入后，必须避免：

- widget 刷新一份、弹框再各自刷一份导致抖动
- 关闭弹框后监听未释放

### 10.3 交互映射冲突

当前 `HabitListItem` 同时支持：

- `open-doc`
- `open-detail`
- `open-calendar`

工作台 widget 需要明确事件映射：

- 打开详情弹框应走统一的“详情打开”语义
- 不能误复用“打开日历”这类带旧语义的事件名而导致后续理解混乱

如果现有事件命名已不准确，允许在本次顺手整理，但范围只限习惯链路。

---

## 十一、测试策略

至少覆盖以下层面：

### 11.1 Widget 渲染

- `habitWeek` widget 在无配置时展示全部习惯
- 指定 `groupId` 后只展示对应分组习惯
- 无习惯时展示空状态

### 11.2 配置持久化

- 打开 widget 配置弹框可选择分组
- 确认后写入 workbench store
- 重新加载工作台后配置仍生效

### 11.3 详情弹框

- 点击习惯项会打开弹框
- 弹框内展示详情主体
- 关闭后可再次打开
- 多次打开关闭不会残留旧内容或重复监听

### 11.4 打卡回流

- 在 widget 列表打卡后，摘要与列表状态更新
- 在详情弹框中触发操作后，详情与列表状态同步

---

## 十二、验收标准

本次完成后，应满足以下标准：

1. 工作台 `habitWeek` widget 不再只是单一周条，而是“周摘要 + 习惯列表”
2. widget 列表项点击后以弹框展示习惯详情
3. 详情内容与交互尽量复用现有习惯详情能力
4. widget 支持分组筛选配置，并写入 workbench JSON
5. 侧栏 `HabitDock` 原有交互不变
6. 不引入新的明显重复实现或状态漂移点
