# 四象限 Tab 设计

## 一、背景

当前插件已经提供 Calendar、Gantt、Project、Pomodoro Stats 等顶层 Tab，以及 Todo Dock 侧边待办视图。Todo Dock 已经具备成熟的事项展示能力，包括：

- 基于 `projectStore` 的过滤与排序
- 事项卡片渲染
- 分组折叠与展开
- 右键菜单与事项详情
- 事项完成、放弃、优先级调整等交互

这次需求不是再做一个新的待办列表实现，而是新增一个独立顶层 Tab，用“四象限”视角重新组织现有事项，并继续复用 Todo 事项卡片和主交互。

---

## 二、目标

### 2.1 目标行为

新增一个桌面端独立顶层 `QuadrantTab`：

- 在插件顶层 Tab 中与 Calendar / Gantt / Project / Pomodoro Stats 同级
- 页面主体为四宫格，每个象限展示一组事项
- 象限内事项卡片与 Todo Dock 保持一致
- 四象限保留“重要 / 紧急”的命名语义
- 四象限内部实际按现有优先级字段映射，不引入新的数据模型

### 2.2 四象限映射

四象限命名保留经典语义，但内部映射改为基于事项优先级：

- 第一象限 `重要且紧急` -> `priority === 'high'`
- 第二象限 `重要不紧急` -> `priority === 'medium'`
- 第三象限 `紧急不重要` -> `priority === 'low'`
- 第四象限 `不重要不紧急` -> `priority === undefined`

这意味着用户看到的是熟悉的四象限视图，但系统底层只依赖当前已有的 `high / medium / low / 未设置` 四档优先级。

### 2.3 非目标

本次不做：

- 不新增“重要”或“紧急”两个独立字段
- 不扩展 Markdown 标记语法
- 不改变移动端导航结构
- 不把 Todo Dock 整套筛选面板复制到四象限 Tab
- 不新写一套独立事项卡片实现

---

## 三、交互设计

### 3.1 顶层入口

`QuadrantTab` 作为新的桌面端 Tab 注册：

- 在 `src/index.ts` 中加入新的 `TAB_TYPES.QUADRANT`
- 使用独立图标和标题
- 可从插件顶栏菜单中打开

它是一个顶层工作视图，不挂在 Todo Dock 内部，也不依赖 Todo Dock 已打开。

### 3.2 页面结构

页面整体结构分为两层：

1. 顶部工具栏
2. 下方四宫格主体

四宫格主体中包含四个象限卡片，每个象限卡片显示：

- 象限标题
- 当前事项数量
- 象限列表内容

每个象限都有自己的空状态。即使只有部分象限有数据，也不应退化为整页空白。

### 3.3 工具栏动作

工具栏只保留与该视图直接相关的动作：

- 刷新
- 全部折叠 / 全部展开
- 更多菜单

“更多菜单”沿用 Todo Dock 的可见性类设置：

- `hideCompleted`
- `hideAbandoned`
- `showLinks`
- `showReminderAndRecurring`

这样四象限与 Todo Dock 在展示偏好上保持一致，但不把四象限做成第二个 Todo Dock。

### 3.4 全局筛选

`QuadrantTab` 顶层仅保留少量全局筛选：

- 分组
- 搜索

这两个条件同时作用到四个象限。

本次不在四象限中提供：

- 日期范围筛选
- 优先级按钮筛选
- 排序规则编辑面板

原因是四象限本身已经按优先级分桶；再把优先级筛选和复杂筛选条搬进去，会明显稀释该视图的核心用途。

### 3.5 象限内事项交互

象限内事项卡片继续沿用 Todo 现有能力，包括但不限于：

- 折叠 / 展开
- 打开文档
- 查看详情
- 标记完成
- 标记放弃
- 修改优先级
- 查看与编辑提醒、重复
- 展示链接

其中最关键的行为是：

- 当用户在四象限中修改事项优先级后，事项应立即离开当前象限
- 刷新后的目标象限应立即出现该事项

这既是用户心智上的核心反馈，也是实现正确性的关键判据。

### 3.6 跨象限拖拽改优先级

四象限页支持桌面端鼠标拖拽，用于“跨象限后改优先级”：

- 事项卡片可作为拖拽源
- 四个象限面板可作为投放目标
- 放手后只更新优先级，不承诺象限内插入位置

落点规则按象限面板命中，不细分到日期分组。拖入后，系统按象限映射目标优先级：

- `重要且紧急` -> `high`
- `重要不紧急` -> `medium`
- `紧急不重要` -> `low`
- `不重要不紧急` -> `undefined`

第四象限继续保留四象限语义名称，但系统行为等价于清除优先级。

本次拖拽交互不包含：

- 象限内手动重排
- 真实插入位置预览
- 移动端拖拽
- 多选拖拽
- 引入 `SortableJS` 等第三方拖拽库

如果拖到与当前优先级一致的象限，则视为 no-op，不触发写回。

## 四、组件与数据流设计

### 4.1 `QuadrantTab` 的职责

新增 `src/tabs/QuadrantTab.vue`，职责限定为：

- 渲染顶层工具栏
- 渲染四宫格布局
- 将分组、搜索、象限过滤参数传给每个列表实例
- 监听现有刷新链路并驱动视图更新

`QuadrantTab` 不直接实现事项卡片，不直接操作块内容，不新增独立数据源。

### 4.2 复用 `TodoSidebar`

四个象限都复用 `TodoSidebar` 作为列表主体，这样可以直接复用：

- 事项卡片
- 分组展示逻辑
- 已完成 / 已放弃分区
- 列表操作与右键菜单
- 与 `projectStore` 的既有协作方式

这次设计的重点是“重新组织列表输入参数”，而不是复制 Todo 列表内部实现。

### 4.3 `TodoSidebar` 需要的最小扩展

为了支撑四象限，`TodoSidebar` 需要两个最小扩展点：

1. 支持“未设置优先级”的过滤能力  
   当前 `priorities: PriorityLevel[]` 只能表达 `high / medium / low`，不能表达 `priority === undefined` 的集合。第四象限需要一个显式开关或等价参数来筛选“未设置优先级”的事项。

2. 支持轻量展示模式  
   四象限中每个列表会被嵌入更小的容器里，需要一个轻量模式来微调空状态文案、容器间距、默认折叠行为或局部标题展示方式。这个模式不改变事项卡片结构，也不分叉主交互。

3. 支持可选的拖拽源事件  
   四象限需要把事项卡片暴露为 drag source，但拖到哪里、落地后做什么，仍由 `QuadrantTab` 决定。`TodoSidebar` 只负责在卡片层发出最小拖拽上下文，例如 `blockId`、`itemId`、当前优先级。

除此之外，不应把 `TodoSidebar` 改造成新的通用页面框架。

### 4.4 数据来源与刷新

四象限继续复用现有数据来源：

- `projectStore`
- `settingsStore`
- `eventBus`
- `BroadcastChannel`

刷新行为与其他 Tab / Dock 保持一致：

- 同上下文刷新通过 `eventBus`
- 跨上下文刷新通过现有 `DATA_REFRESH_CHANNEL`

`QuadrantTab` 需要像 `DesktopTodoDock` 一样接入这条刷新链路，避免出现 Dock 更新但顶层 Tab 不刷新的分叉行为。

### 4.5 排序语义

四象限内事项顺序继续沿用 `settingsStore.todoDock.sortRules`。

原因：

- 保持 Todo Dock 与四象限之间的顺序一致性
- 不额外引入第二套排序偏好
- 复用现有 `projectStore` 排序逻辑

四象限负责“分桶”，桶内排序仍由现有 Todo 排序规则决定。

### 4.6 拖拽后的数据写回

跨象限拖拽完成后，`QuadrantTab` 不直接修改本地数组，也不手动移动 DOM，而是调用现有 `updateBlockPriority(blockId, targetPriority)` 写回文档。

写回成功后，四象限走现有刷新链路重新拉取数据，让事项按过滤规则自然归位到目标象限。这样可以保证拖拽、右键菜单修改优先级、详情弹框修改优先级共用同一条写回路径。

写回失败时，当前视图不做本地迁移，并给出失败提示。

## 五、实现边界

### 5.1 常量与入口

需要修改：

- `src/constants.ts`
- `src/index.ts`
- `src/i18n/zh_CN.json`
- `src/i18n/en_US.json`

职责：

- 新增 `TAB_TYPES.QUADRANT`
- 为顶层 Tab 注册新视图
- 在顶栏菜单中加入入口
- 配置图标、标题和国际化文案

### 5.2 新增 `QuadrantTab.vue`

需要新增：

- `src/tabs/QuadrantTab.vue`

职责：

- 搭建工具栏
- 搭建四宫格布局
- 维护顶层分组与搜索状态
- 为四个象限组装不同的过滤参数
- 接入现有刷新通道

### 5.3 调整 `TodoSidebar.vue`

需要修改：

- `src/components/todo/TodoSidebar.vue`

职责：

- 支持第四象限所需的“未设置优先级”过滤能力
- 支持四象限所需的轻量显示模式
- 支持可选的事项拖拽源事件发射
- 保持现有 Todo Dock 行为不变

原则是新增可选能力，不破坏现有 Dock 的调用方式。

### 5.4 过滤与 store 协作

如果当前 `projectStore` 的过滤 API 不能自然表达“未设置优先级”，则需要最小扩展：

- `src/stores/projectStore.ts`

扩展目标仅限于表达四象限过滤条件，不顺带重构整个过滤体系。

### 5.5 测试文件

预计需要新增或调整：

- `test/tabs/QuadrantTab.test.ts`
- `test/components/todo/TodoSidebar.test.ts` 或同等粒度的拖拽测试文件
- `test/stores/projectStore.test.ts`

新增拖拽后，测试重点是：

- 象限 key 到目标优先级的映射是否正确
- drop 成功后是否触发优先级写回与刷新
- 同优先级 drop 是否 no-op
- `TodoSidebar` 是否能发出最小拖拽上下文

如 `TodoSidebar` 的新参数带来组件行为变化，也可以补充该组件对应测试，但范围只覆盖四象限新增能力。

## 六、错误处理与兼容性

### 6.1 未设置优先级事项

第四象限的事项定义明确为：

- `item.priority === undefined`

不应通过“排除 high / medium / low 后剩余事项”这种隐式方式推导，以免未来字段扩展后语义漂移。

### 6.2 老数据兼容

因为本次完全复用现有优先级字段：

- 旧数据无需迁移
- 旧用户无需修改文法
- 未设置优先级的历史事项自动落入第四象限

### 6.3 象限内容为空

单个象限为空时：

- 展示该象限空状态
- 其他象限正常工作

整个视图只有在四个象限都无数据时，才表现为全局空视图。

### 6.4 刷新与视图迁移

事项在以下场景中都必须正确迁移象限：

- 修改优先级
- 刷新项目数据
- 切换分组
- 改变搜索词
- 切换隐藏已完成 / 已放弃设置

如果某个象限的数据刷新失败，不应导致整个 Tab 崩溃；行为上沿用现有列表的容错策略即可。

## 七、测试设计

### 7.1 过滤逻辑测试

重点覆盖：

- `high` 事项进入第一象限
- `medium` 事项进入第二象限
- `low` 事项进入第三象限
- `undefined` 优先级事项进入第四象限
- 分组和搜索条件会同时作用到四个象限

如果过滤逻辑在 `projectStore` 层实现，这些测试优先落在 store 单测里。

### 7.2 组件测试

`QuadrantTab` 组件测试至少覆盖：

- 会渲染四个象限容器
- 每个象限向 `TodoSidebar` 传入正确的优先级过滤条件
- 第四象限会传入“未设置优先级”过滤标记
- 工具栏的刷新按钮可以触发 `projectStore.refresh`

### 7.3 交互回归测试

至少覆盖一个核心回归场景：

- 某事项初始位于一个象限
- 修改其优先级后
- 该事项从原象限移除，并出现在目标象限

这个测试可以在组件层或更低层实现，但必须能证明“象限迁移”成立。

### 7.4 现有行为回归

需要回归确认：

- Todo Dock 现有行为不变
- `hideCompleted / hideAbandoned / showLinks / showReminderAndRecurring` 在四象限中仍然生效
- 右键菜单、详情弹框、链接展示没有因为新容器而失效

---

## 八、实施建议

建议按以下顺序实施：

1. 先补 `TAB_TYPES`、i18n、顶层 Tab 注册入口
2. 再补 `TodoSidebar` 或 `projectStore` 对“未设置优先级”的表达能力
3. 新建 `QuadrantTab.vue` 接线四宫格
4. 最后补测试并回归顶层菜单打开流程

这个顺序可以先把数据表达问题解决，再接 UI，避免四象限页面写完后发现第四象限筛选条件无法准确表达。

---

## 九、结论

本方案在不引入新数据模型的前提下，为插件新增一个独立顶层四象限 Tab。它保留“重要 / 紧急”的视图语义，但内部严格复用现有优先级字段完成映射；列表主体继续复用 `TodoSidebar` 和既有事项卡片，从而保证交互一致、实现风险可控、维护成本较低。实现上的关键不在于重写待办列表，而在于给现有过滤能力补上“未设置优先级”这条缺失表达，并将该能力稳定接入一个新的四宫格容器。
