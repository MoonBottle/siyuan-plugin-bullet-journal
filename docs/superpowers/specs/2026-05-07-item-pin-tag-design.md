# Todo Item 置顶与标签设计

## 一、功能概述

本设计为 Todo item 增加两个面向日常整理的轻量能力：

1. **置顶**：使用 `📌` 将某个事项固定排在 Todo Dock 列表前部
2. **标签**：复用思源原生 `#标签` 语义，为事项提供分类、搜索和筛选能力

本次范围严格限定在 `item`，不扩展到 `task`。置顶采用 Markdown 行内持久化标记，标签沿用思源原生文本标签写法，由插件解析后结构化使用。

### 1.1 设计目标

- **保持 Markdown 为权威源**：置顶状态写回事项原始行，刷新和跨端后仍然生效
- **遵循现有 parser 架构**：沿用 `priority/reminder/recurring` 的“行内标记 -> 结构字段”模式
- **标签不另起体系**：直接读取思源原生 `#标签`，避免引入额外的 tag DSL
- **最小侵入 Todo Dock**：优先补齐排序、展示、搜索、筛选，不先做复杂标签编辑器
- **兼容旧数据**：没有 `📌` 或标签的事项在升级后行为不变

## 二、需求范围

### 2.1 已有能力

当前系统已经支持：

- item 级日期、时间、多日期展开
- item 状态标记：`#done` / `#已完成` / `#abandoned` / `#已放弃` / `✅` / `❌`
- item 优先级、提醒、重复规则解析
- Todo Dock 搜索、日期过滤、优先级过滤、多规则排序
- UI 侧对 item 的局部写回能力，例如状态、提醒、重复等相关操作

### 2.2 本次新增能力

本次新增：

- item 行内 `📌` 标记解析为 `item.pinned`
- item 行内业务标签 `#标签` 解析为 `item.tags`
- Todo Dock 顶部新增独立的“已置顶”分组
- Todo Dock 事项卡片展示图钉状态和标签
- Todo Dock 搜索支持命中标签
- Todo Dock 主搜索框下方新增独立标签搜索框
- 标签搜索框支持多选过滤
- 标签候选项基于当前解析出的业务标签聚合生成
- 用户文档补充 `📌` 和标签语义说明

### 2.3 非目标

本次不做：

- 不支持 `task` 级置顶或标签
- 不新增独立标签管理页
- 不做标签 UI 写回编辑
- 不支持手动维护标签字典或独立标签管理页
- 不引入块属性作为第一版置顶存储方式
- 不处理层级标签、标签别名、标签规范化中心

## 三、Markdown 语法设计

### 3.1 置顶标记

置顶使用行内标记 `📌`：

```markdown
修复登录回调 @2026-05-07 📌
```

规则：

- `📌` 只在 item 行中生效
- `📌` 可出现在行内任意位置，不依赖固定顺序
- UI 写回时统一使用 `📌`，不引入 `#pin` 或块属性双轨制

### 3.2 标签语法

标签直接复用思源原生 `#标签` 语法：

```markdown
修复登录回调 @2026-05-07 📌 #前端 #认证
联系客户确认报价 @2026-05-08 #客户A #销售跟进
```

规则：

- 标签解析目标为 item 行中的原生标签 token
- 解析结果落到 `item.tags`
- 存储时不保留前缀 `#`，例如 `['前端', '认证']`

### 3.3 保留标签

以下标签继续保留现有语义，不计入业务标签：

- `#done`
- `#已完成`
- `#abandoned`
- `#已放弃`
- `#task`
- `#任务`

例如：

```markdown
整理需求池 @2026-05-09 ✅ #产品
```

解析结果应为：

- `status = completed`
- `tags = ['产品']`

### 3.4 推荐书写顺序

推荐按以下顺序书写，但解析不强依赖顺序：

```text
内容 @日期 [⏰提醒] [🔁重复规则] [🔥优先级] [📌] [状态标签] [业务标签...]
```

## 四、数据模型设计

在 [src/types/models.ts](/C:/dev/projects/open-source/siyuan-plugin-bullet-journal/src/types/models.ts) 的 `Item` 上新增两个可选字段：

- `pinned?: boolean`
- `tags?: string[]`

语义如下：

- `pinned`：当前事项行中是否存在 `📌`
- `tags`：当前事项行中解析出的业务标签数组，不包含 `#`

边界约束：

- 多日期事项展开后，每个派生 item 共用同一组 `pinned/tags`
- `tags` 第一版不承载层级语义，也不区分标签来源
- 无标签时可为 `undefined` 或空数组，具体实现保持与项目现有字段风格一致

## 五、Parser 设计

### 5.1 职责拆分

建议延续现有 parser 拆分方式，新增两个小模块：

- `src/parser/pinParser.ts`
  - `parsePinnedFromLine(line): boolean`
  - `stripPinnedMarker(text): string`
- `src/parser/tagParser.ts`
  - `parseTagsFromLine(line): string[]`
  - `stripTagsFromLine(text): string`
  - `isReservedTag(tag): boolean`

`lineParser.ts` 继续作为 item 行解析的 orchestration 层，不再额外堆积新的 token 细节。

### 5.2 parseItemLine 处理顺序

建议在 `parseItemLine()` 中采用以下顺序：

1. 校验当前行是否是 item 行
2. 解析 reminder、priority、repeatRule、status
3. 解析 `pinned`
4. 解析业务标签 `tags`
5. 提取日期时间表达式并展开多日期
6. 清洗内容：
   - 去日期与时间
   - 去状态标记
   - 去任务列表标记
   - 去 reminder / priority / recurring
   - 去 `📌`
   - 去业务标签
7. 解析块引用与链接
8. 为最终生成的每个 item 注入 `pinned/tags`

### 5.3 标签解析规则

`tagParser` 的核心规则：

- 提取符合思源原生标签形式的 `#xxx`
- 使用保留标签名单剔除状态和任务标记
- 对业务标签去重并过滤空值
- `stripTagsFromLine()` 仅移除业务标签，不移除保留标签

这样可以保证：

- 状态判断仍由现有逻辑负责
- 任务标记不会误进入 `item.tags`
- 标签语义与正文清洗解耦

### 5.4 内容清洗结果

`item.content` 应保留干净的正文，不再混入：

- 日期时间表达式
- `📌`
- 业务标签
- reminder / priority / recurring token
- 状态标签

例如：

```markdown
修复登录回调 @2026-05-07 🔥 📌 #done #前端 #认证
```

应得到：

- `content = '修复登录回调'`
- `priority = high`
- `pinned = true`
- `status = completed`
- `tags = ['前端', '认证']`

### 5.5 兼容性要求

- 未包含 `📌` 和标签的历史 item 不受影响
- `📌` 出现在任意位置都可识别
- 多日期展开不改变 `pinned/tags`
- 标签解析失败时，事项解析整体不能失败

## 六、Todo Dock 行为设计

### 6.1 分组与排序

Todo Dock 不再只用“置顶优先排序”表达 `pinned`，而是显式拆成两个列表分组：

1. `已置顶`
2. `全部事项` 或当前过滤结果列表中的常规事项

分组规则：

1. `item.pinned === true` 的事项进入顶部“已置顶”分组
2. `item.pinned !== true` 的事项进入下方常规分组
3. 两个分组内部继续按现有排序规则排序

这样 `pinned` 既有明确的信息架构，也不会和普通排序规则混在一起导致用户误判“为什么这条一直在最前面”。

对数据层的影响：

- `projectStore` 仍保留现有多规则排序能力
- 但 Todo Dock 消费数据时需要额外产出两个集合，或产出一个带分组信息的视图模型
- `compareTodoItems()` 仍只负责分组内排序，不再承担“是否置顶”的分层语义

### 6.2 搜索

搜索框需要同时命中：

- `item.content`
- 项目名 / 任务名
- `item.tags`

兼容行为：

- 搜索 `前端` 可以命中 `#前端`
- 搜索 `#前端` 也可以命中 `#前端`

实现上建议在搜索时对查询值做一次 `#` 前缀兼容处理，而不是要求用户记忆固定输入格式。

### 6.3 标签搜索与筛选

Todo Dock 在主搜索框下方新增一个独立的标签搜索框，用于基于已解析业务标签进行筛选。

交互规则：

- 标签搜索框位于主搜索框下方，属于同一过滤区
- 候选项来源于当前 Todo 数据中解析出的 `item.tags`
- 候选项在运行时聚合生成，不依赖手工维护标签字典
- 支持多选
- 已选择标签以紧凑 token 形式显示在输入框内或输入框下方

筛选语义：

- 默认采用 **OR 语义**：事项命中任一已选标签即可显示
- 这样更符合“按主题/领域快速收窄列表”的使用预期
- 若后续出现明确需求，再单独设计 AND 语义或切换开关

搜索体验：

- 输入关键字时，对聚合出的标签候选项进行本地过滤
- 既支持输入 `前端`，也支持输入 `#前端`
- 候选项可附带数量信息，例如 `前端 (12)`，便于用户判断筛选价值

聚合边界：

- 仅聚合 parser 识别出的业务标签
- 状态标签和任务标签不进入候选项
- 候选项去重后按“数量降序 + 名称升序”排序

### 6.4 展示

Todo Dock 顶部增加独立分组标题，例如“已置顶”。

展示要求：

- 仅当存在置顶事项时显示该分组
- “已置顶”分组位于常规事项列表上方
- 分组标题使用 Dock 内现有的弱标题层级，不做强卡片化包裹
- 置顶分组和常规分组之间保留明确但克制的垂直间距

事项卡片建议新增两个展示元素：

- 图钉图标：标识 `item.pinned === true`
- 标签区：展示解析后的业务标签

视觉要求：

- 图钉作为状态提示，不喧宾夺主
- 标签使用紧凑弱强调样式，避免压过主内容
- 桌面端与移动端共用数据语义，允许在密度上做差异化展示
- 点击事项卡片上的标签可作为快捷操作，将该标签加入标签搜索框当前选中值

### 6.5 写回策略

第一版只对置顶提供 UI 写回：

- 用户点击图钉按钮
- 插件修改事项原始 Markdown 行，插入或移除 `📌`
- 写回后触发现有刷新链路，重新解析得到 `item.pinned`

标签第一版只读，不提供 UI 编辑写回。原因：

- 标签回写涉及多个 token 的增删改和格式保留
- 其复杂度明显高于单一 `📌` 标记
- 当前阶段优先保证 parser、搜索、筛选和展示能力稳定

## 七、异常处理与边界

### 7.1 降级原则

解析失败时不得影响 item 主体展示：

- `📌` 识别失败时，按未置顶处理
- 标签解析失败时，`tags` 退化为空，但正文仍需可见

### 7.2 去重与清理

- 重复业务标签需要去重
- 空标签和非法空白标签不写入结果
- 保留标签优先级高于业务标签

### 7.3 暂不处理的复杂场景

本次不对以下情况做精细语义支持：

- URL 或代码片段中的 `#`
- 层级标签
- 标签别名
- 自动标签规范化

这类场景后续若成为真实需求，再单独补设计，不在第一版提前复杂化 parser。

## 十一、最终决策

本设计最终决策如下：

- **范围**：仅支持 `item`
- **置顶持久化**：使用 Markdown 行内 `📌`
- **标签来源**：复用思源原生 `#标签`
- **第一版标签能力**：解析、展示、主搜索命中、独立标签搜索框、多选筛选
- **第一版置顶能力**：解析、分组展示、分组内排序、UI 写回
- **Parser 结构**：新增 `pinParser.ts` 和 `tagParser.ts`，由 `lineParser.ts` 编排

该方案与当前仓库的数据流和 parser 风格一致，能以较小代价补齐 Todo Dock 的高频整理能力，同时把标签写回这种复杂问题留到下一轮。
