# 数据同步与刷新协调重构设计

## 背景

当前插件的数据同步链路已经覆盖了大多数场景：

- 插件启动时全量加载项目数据
- 思源 `ws-main` 事件驱动后续刷新
- 本地块内容修改通过 `LOCAL_DATA_MUTATED` 触发刷新
- 跨上下文通过 `BroadcastChannel` 同步刷新请求

这套机制已经具备可用性，但同步职责分散在 `index.ts`、`projectStore`、多个 Tab/Dock/移动端面板中，导致刷新行为逐渐变得难以推断、难以验证，也开始出现用户可感知的问题。

其中最明显的问题是：`projectStore.ts` 中全量刷新会先执行 `this.projects = []`，再逐个 `push` 新项目。视图会在“数据被清空”到“新数据逐步回填”的时间窗口里进入短暂空状态，表现为明显闪烁。

## 现状问题

### 1. 刷新执行权分散

当前 `DATA_REFRESH` 事件发出后，并不是由单一入口执行刷新，而是各个视图各自监听事件，再分别调用 `projectStore.refresh()`。

结果是：

- 同一轮刷新请求会被多个视图重复消费
- 刷新行为依赖“哪个视图先执行”
- 视图层混入了大量同步逻辑和 settings patch 逻辑
- 新增一个视图时，容易复制出另一套刷新接线

### 2. 刷新中的新变更可能被吞掉

`projectStore.refresh()` 当前通过 `refreshing` 标志防止并发刷新。如果刷新过程中又来了新的变更请求，新的调用会直接返回。

这意味着：

- 新的脏文档可能已经进入 `dirtyDocTracker`
- 新的刷新事件也已经广播出去
- 但本轮刷新结束后，没有补偿机制确保这些新变更一定再跑一轮

这属于一致性风险，不只是性能问题。

### 3. 触发语义不统一

当前存在三类刷新触发路径：

- `ws-main` 事件触发
- 本地块修改后发 `LOCAL_DATA_MUTATED`
- 业务组件或服务直接发 `DATA_REFRESH`

这些路径并没有统一的语义边界，容易出现：

- 某些场景等 `ws-main`
- 某些场景本地写成功后立即刷新
- 某些场景事件和手动刷新叠加，产生重复刷新

### 4. 全量刷新导致闪烁

当前全量刷新流程会先清空 `projects`，再逐步回填。

这会导致：

- 所有依赖 `projects` 的视图瞬间看到空数组
- 日历、待办、习惯、工作台等视图出现空态闪烁
- 正在使用中的用户界面在后台刷新时出现不稳定的视觉跳变

### 5. 定向刷新能力不完整

当前定向刷新主要支持“替换已有项目”或“新增项目”，但对以下场景支持不足：

- 文档删除后应从结果集中移除
- 文档移动后不再匹配当前扫描范围
- 扫描目录或 `scanMode` 变更后，已有项目需要重新归类或移除

这意味着很多边界场景仍然需要回退到全量刷新思路。

## 目标

本次重构需要同时达成以下目标：

1. 刷新执行权收口到单一协调入口。
2. 刷新过程具备串行化和补偿能力，不吞掉刷新中的新变更。
3. 全量刷新不再引发视图闪烁。
4. 刷新原因和刷新决策显式化，减少隐式耦合。
5. 各视图不再直接承担 `projectStore.refresh()` 的调度责任。
6. 保留现有定向刷新收益，并补齐删除、移动、配置变更等边界场景。

## 核心判断

这次问题的根因不是单一 bug，而是“刷新触发、刷新决策、刷新执行、视图响应”四层职责混在一起。

因此本次设计不采用继续堆补丁的方式，而采用以下原则：

- 触发统一进入刷新协调器
- 协调器负责聚合和排队
- `projectStore` 只负责执行刷新，不负责调度
- 视图只消费 store 状态，不直接驱动 refresh

## 信息架构与职责边界

### 1. `index.ts`

职责：

- 接入外部事件源
- 把 `ws-main`、`LOCAL_DATA_MUTATED`、手动刷新、BroadcastChannel 等信号转成统一的刷新请求
- 在插件生命周期内创建并持有刷新协调器

不再承担的职责：

- 不再通过 `DATA_REFRESH` 让各视图自己去调用 `projectStore.refresh()`

### 2. 刷新协调器

新增一个显式的刷新协调模块，作为本次重构的核心。

职责：

- 接收刷新请求
- 聚合脏文档集合
- 判断是否需要全量刷新、定向刷新或仅同步 settings
- 串行执行刷新
- 在刷新结束后检查是否需要补跑下一轮
- 在刷新稳定完成后统一发出 `DATA_REFRESHED`

不负责：

- 具体解析项目
- 直接维护 UI 状态
- 直接承担业务逻辑

### 3. `projectStore`

保留职责：

- `loadProjects`
- `refreshFull`
- `refreshDirtyDocs`
- 项目数组落地与数据派生

新增约束：

- store 不再作为并发刷新调度器
- store 不再通过“直接返回”吞掉刷新中的后续请求
- store 的全量刷新必须支持“后台构建新数组，完成后一次性提交”

### 4. 各 Tab/Dock/移动端面板

职责收敛为：

- 监听 settings payload 并做必要 patch
- 响应用户手动刷新操作，向协调器发请求
- 读取 `projectStore` 的响应式数据渲染界面
- 视需要显示 `refreshing` 轻量状态

不再承担的职责：

- 不再在 `handleDataRefresh()` 内直接执行 `projectStore.refresh()`

## 刷新请求模型

刷新请求需要从“事件名驱动”改为“显式意图驱动”。

建议至少区分三类请求：

### 1. `settings-only`

含义：

- 仅同步设置变化
- 不需要重新扫描项目数据

典型场景：

- 日历默认视图
- Todo 排序规则
- 仅影响显示层的配置项

### 2. `directed`

含义：

- 已知受影响文档集合
- 优先尝试定向刷新

典型场景：

- `savedoc`
- `transactions/updateAttrs`
- 本地单块修改提醒、重复、置顶等

### 3. `full`

含义：

- 当前变更影响范围无法局部确认
- 或者当前局部状态已不足以保证正确性

典型场景：

- 目录配置变更
- `scanMode` 变更
- `moveDoc`
- `removeDoc`
- 无法提取 rootID 的事务
- 定向刷新过程中发生无法判定范围的异常

## 刷新决策规则

协调器需要明确决策优先级：

1. 若存在 `full` 请求，则本轮刷新强制全量。
2. 否则若存在 `directed` 请求且有脏文档，执行定向刷新。
3. 否则若仅有 `settings-only` 请求，只同步 settings，不刷新项目数据。

额外规则：

- 全量请求优先级高于定向请求
- 同一轮内多个定向请求的 docId 应合并去重
- 定向刷新执行失败时，可升级为全量刷新
- settings-only 请求不能隐式触发项目重扫

## 刷新流程设计

### 1. 请求进入

所有外部刷新信号统一进入协调器，协调器维护以下聚合状态：

- `isRunning`
- `pendingFullRefresh`
- `pendingDirtyDocIds`
- `pendingSettingsPayload`

### 2. 启动执行

- 如果当前没有 refresh 在执行，协调器立即启动一轮刷新
- 如果已经在执行，则只更新 pending 状态，不丢弃请求

### 3. 执行策略

执行时：

- 先冻结当前 pending 状态作为本轮快照
- 根据快照决定 full / directed / settings-only
- 执行完成后检查运行期间是否新增 pending 请求
- 如果有，则立即进入下一轮

### 4. 完成通知

只有当本轮刷新稳定完成后，才统一发出 `DATA_REFRESHED`。

提醒服务、移动端通知重建等后续动作继续挂在 `DATA_REFRESHED` 后执行，而不是散落在 `scheduleRefresh()` 之类的预刷新阶段。

## 无闪烁全量刷新设计

这是本次必须显式解决的问题。

### 当前问题

现有流程：

1. `this.projects = []`
2. 逐个解析项目
3. 逐个 `push`

这会让视图在刷新中看到短暂空态。

### 新流程

全量刷新改为：

1. 保留当前 `projects` 作为稳定旧数据
2. 在后台解析得到完整的 `nextProjects`
3. 全部准备好之后，一次性提交替换
4. 再统一做后续派生动作与完成通知

### 结果

这样可以保证：

- 刷新过程中界面始终有旧数据可显示
- 不再出现“先清空后重建”的视觉闪烁
- 后台刷新期间用户看到的是稳定界面 + 轻量刷新状态，而不是空界面

### UI 配套要求

- 现有 `refreshing` 状态保留，用于显示轻量 busy 状态
- 不应在全量刷新期间主动切换到空态文案
- 若局部组件需要 loading 指示，应使用遮罩、按钮态或角标，而不是销毁主要内容区域

## 定向刷新补强

定向刷新不应只支持 replace/add，还需要补齐 remove 语义。

### 需要覆盖的情况

- 文档已删除
- 文档已移出当前扫描范围
- 文档路径变化导致分组匹配变化
- 配置变更导致原有项目不应继续存在于当前结果集

### 处理原则

- 当定向刷新发现目标文档已不存在，或不再满足扫描条件时，应从 `projects` 中移除对应项目
- 当定向刷新无法安全判定 remove/replace/add 时，升级为全量刷新

## 视图层改造范围

本次需要改造所有直接在刷新事件中调用 `projectStore.refresh()` 的视图。

典型包括但不限于：

- `src/tabs/CalendarTab.vue`
- `src/tabs/DesktopTodoDock.vue`
- `src/tabs/GanttTab.vue`
- `src/tabs/ProjectTab.vue`
- `src/tabs/WorkbenchTab.vue`
- `src/tabs/PomodoroDock.vue`
- `src/tabs/PomodoroStatsTab.vue`
- `src/tabs/AiChatDock.vue`
- `src/mobile/panels/MobileTodoPanel.vue`
- `src/mobile/panels/MobileHabitPanel.vue`
- 相关 workbench 习惯视图与细节弹层

改造目标：

- 这些视图不再直接执行 refresh
- 仅保留 settings 同步和界面更新职责
- 手动刷新按钮改为“向协调器发刷新请求”

## 兼容与迁移策略

本次重构不要求一次性推翻现有事件名，但要求逐步收口语义。

建议迁移策略：

1. 第一阶段保留现有 `DATA_REFRESH` 事件名，但内部改为由协调器消费和执行。
2. 第二阶段移除视图内对 `projectStore.refresh()` 的直接调用。
3. 第三阶段将各业务侧“直接 emit DATA_REFRESH”的路径改为更显式的刷新请求接口。

这样可以控制改动风险，避免一次性重写所有同步路径。

## 测试策略

至少需要覆盖以下验证点：

### 1. 全量刷新不闪烁

- 全量刷新过程中，`projects` 不应先变成空数组
- 旧数据应持续可读，直到新数据一次性替换完成

### 2. 刷新补偿

- refresh 进行中再次收到刷新请求
- 当前 refresh 结束后应自动补跑下一轮

### 3. 定向刷新路径

- `savedoc` + `rootID` 能触发定向刷新
- `transactions/updateAttrs` 能正确提取 `rootIDs`
- 多个脏文档能合并去重

### 4. 全量刷新升级

- `moveDoc` / `removeDoc` / 配置变更能强制走全量
- 定向刷新异常时可升级为全量

### 5. settings-only 同步

- 纯 settings 广播不会触发项目重扫
- 视图能收到 settings patch 并正常渲染

### 6. 多视图并发监听

- 多个视图同时收到刷新信号时，只执行一次真实 refresh

## 建议实施顺序

### 第一步：消除全量刷新闪烁

- 重写 `refreshFull()` 的数据提交方式
- 先解析到 `nextProjects`
- 再一次性替换 `projects`

这是用户最直接可感知的问题，应优先解决。

### 第二步：引入 refresh queue / pending 机制

- 解决刷新中的新变更被吞掉的问题
- 建立串行刷新与补偿重跑能力

### 第三步：接入刷新协调器

- 把刷新请求统一收口
- 让 `index.ts` 和跨上下文入口都走同一条协调路径

### 第四步：收回视图执行权

- 去掉各视图中的直接 `projectStore.refresh()`
- 改为请求协调器刷新

### 第五步：细化刷新原因类型

- 区分 full / directed / settings-only
- 减少不必要重扫

## 验收标准

完成后应满足以下标准：

1. 全量刷新期间主要视图不再出现空态闪烁。
2. 刷新执行权收敛到单一协调入口。
3. 刷新中到来的新请求不会被静默丢弃。
4. 多视图同时在线时，同一轮刷新只执行一次真实数据刷新。
5. 纯 settings 变更不会误触发项目重扫。
6. 删除、移动、配置变更等边界场景能得到正确处理。
7. 提醒调度与移动端通知同步仍然只在 `DATA_REFRESHED` 后执行。

## 实施边界

本次设计聚焦于数据同步与刷新协调重构，不涉及以下内容：

- 解析器语法能力扩展
- 任务/事项数据模型调整
- 视图功能新增
- MCP 能力扩展

如果后续需要进一步优化解析性能、缓存策略或差量合并算法，应作为后续独立设计继续推进。
