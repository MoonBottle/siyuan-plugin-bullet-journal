# 工作日重复事项节假日推进设计

## 一、背景

当前插件对重复规则 `workday` 的处理，仅按“跳过周六、周日”计算下一次日期。这会导致中国法定节假日场景下日期推进错误。

已确认的用户案例：

- 当前事项日期：`2026-04-30`
- 重复规则：`🔁工作日`
- 在 `2026-04-30` 完成后，系统当前会创建 `2026-05-01`
- 但 `2026-05-01` 是中国劳动节假期，正确的下一次日期应为 `2026-05-06`

本设计仅修复 `workday` 规则的日期推进，不扩展到提醒、UI 提示、其他重复规则或其他日期能力。

## 二、目标与非目标

### 2.1 目标

- 让 `workday` 重复规则基于中国法定节假日和补班日推进日期
- 离线状态下仍可正常计算工作日
- 运行时可从远程节假日数据源刷新，刷新成功后覆盖当前内存数据
- 远程拉取失败时不影响现有功能可用性

### 2.2 非目标

- 不修改 `daily` / `weekly` / `monthly` / `yearly` 的推进逻辑
- 不修改重复规则解析语法、文案和 UI 配置项
- 不修改 Todo / Item 详情中“下一次发生时间”的展示逻辑
- 不引入完整通用日历库，不扩展到农历、节气或更多日期能力

## 三、方案选择

本次采用以下方案：

- **仓库内置一份节假日兜底数据**
- **插件运行时可选远程刷新**
- **远程拉取成功后覆盖内存数据，并写入本地缓存**
- **远程拉取失败时继续使用本地缓存；缓存不可用时退回仓库内置数据**

不采用纯远程方案，因为离线和远程失败场景会直接影响日期计算可用性。

不采用 `tyme4ts` 等通用日历库，因为当前需求仅限中国工作日判定，引入完整日历域模型会扩大依赖面和维护成本。

## 四、数据来源与内部格式

### 4.1 上游来源

远程数据源采用 `china-holiday-calender` 仓库提供的 `holidayAPI.json`。

上游数据中，我们只关心两类信息：

- 法定放假日期区间
- 补班日期列表

### 4.2 仓库内置数据

仓库新增一份内置节假日数据文件，作为离线和首次启动的兜底来源。

为降低运行时转换复杂度，内置文件不直接保存上游原始结构，而转换为本项目内部格式：

```ts
interface ChinaWorkdayCalendarData {
  holidays: string[]
  workdays: string[]
}
```

- `holidays`: 所有法定放假日期，格式为 `YYYY-MM-DD`
- `workdays`: 所有补班日期，格式为 `YYYY-MM-DD`

### 4.3 本地缓存

插件运行时将远程拉取并转换后的数据持久化为本地缓存文件，结构与内置数据保持一致。

缓存用途：

- 下次启动优先使用
- 减少每次启动对远程数据的依赖
- 在远程失败时继续提供最近一次成功刷新的数据

## 五、模块设计

新增 `src/services/chinaWorkdayService.ts`，职责如下：

- 加载仓库内置节假日数据
- 读取本地缓存数据
- 拉取远程 `holidayAPI.json`
- 将上游原始结构转换为内部格式
- 提供工作日判定和下一工作日计算能力

对外暴露的最小接口：

```ts
function isChinaWorkday(date: string): boolean
function getNextChinaWorkday(date: string): string
function initializeChinaWorkdayCalendar(): Promise<void>
```

职责边界：

- `chinaWorkdayService` 负责“给定日期是不是中国工作日”
- `recurringParser` / 重复事项相关逻辑只负责“何时调用工作日判定”
- 远程刷新属于后台初始化行为，不进入 `getNextOccurrenceDate` 的同步主路径

## 六、工作日判定规则

对于某个 `YYYY-MM-DD` 日期，判定顺序如下：

1. 如果命中 `workdays`，直接判定为工作日
2. 如果命中 `holidays`，直接判定为非工作日
3. 如果都未命中，则按普通周历回退：
   - 周一到周五：工作日
   - 周六、周日：非工作日

这保证了补班日优先级高于周末规则，法定放假优先级高于普通工作日规则。

## 七、启动与刷新流程

### 7.1 启动读取顺序

插件启动后，`chinaWorkdayService` 的数据加载顺序如下：

1. 优先读取本地缓存
2. 若缓存不存在或损坏，则读取仓库内置数据
3. 加载完成后立即可用于 `workday` 日期计算
4. 然后后台异步尝试远程刷新

### 7.2 远程刷新流程

后台刷新流程如下：

1. 请求远程 `holidayAPI.json`
2. 校验必要字段是否存在且可转换
3. 转换为内部 `holidays/workdays` 格式
4. 用新数据覆盖当前内存数据
5. 写入本地缓存

### 7.3 失败处理

- 远程请求失败：保留当前内存数据，不中断主流程
- 远程数据格式异常：丢弃本次刷新结果，保留当前内存数据
- 本地缓存损坏：忽略缓存，回退内置数据

核心要求：**无论远程是否可达，`workday` 日期计算始终必须返回结果。**

## 八、现有代码改动点

### 8.1 新增文件

- `src/services/chinaWorkdayService.ts`
- 一份仓库内置节假日数据文件

### 8.2 修改文件

- `src/parser/recurringParser.ts`
  - 将 `workday` 分支从“跳过周末”改为调用中国工作日服务
- `src/index.ts`

## 九、行为示例

### 9.1 法定假期跳过

输入：

- 当前日期：`2026-04-30`
- 规则：`workday`

输出：

- 下一次日期：`2026-05-06`

原因：

- `2026-05-01` 至 `2026-05-05` 为劳动节假期

### 9.2 补班周六命中

输入：

- 当前日期：`2026-05-08`
- 规则：`workday`

输出：

- 下一次日期：`2026-05-09`

原因：

- `2026-05-09` 虽为周六，但属于补班日

### 9.3 普通周末回退

输入：

- 当前日期：普通非节假日周五
- 规则：`workday`

输出：

- 下一次日期：下周一

## 十一、验收标准

- [ ] `🔁工作日` 不再仅按周末跳过，而是基于中国法定节假日和补班日推进
- [ ] `2026-04-30` 的下一工作日正确计算为 `2026-05-06`
- [ ] `2026-05-08` 的下一工作日正确计算为 `2026-05-09`
- [ ] 离线状态下仍可计算工作日
- [ ] 远程刷新成功后可覆盖旧数据
- [ ] 远程刷新失败不影响重复事项日期推进
- [ ] 缓存损坏时可退回内置数据
- [ ] 不影响其他重复规则的现有行为

## 十二、决策记录

- 只修复 `workday` 的日期推进，不扩散到 UI 提示和提醒逻辑
- 数据策略采用“内置兜底 + 运行时远程刷新 + 本地缓存”
- 远程拉取成功后覆盖内存并写缓存；失败则继续使用现有可用数据
- 不引入完整通用日历库，保持实现围绕中国工作日判定最小化
