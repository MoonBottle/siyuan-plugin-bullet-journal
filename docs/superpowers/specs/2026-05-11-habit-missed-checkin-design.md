# 习惯未打卡记录设计

## 背景

当前习惯体系已经支持“已打卡”写入与展示，但还缺少两个能力：

- 用户无法在某一天显式记录“今天未完成”
- 用户为历史日期补打卡时，记录行缺少明确的按日期插入规则

这导致月历只能区分“有完成记录”和“无记录”，不能表达“主动标记未完成”；同时，补打卡若总是追加到末尾，会破坏习惯定义下打卡日志的时间顺序，降低文档可读性。

本次设计为习惯详情月历补充未打卡记录、重置操作与补打卡插入规则，并要求 parser、domain 统计和日志展示同步适配。

## 目标

- 为习惯详情月历增加“未打卡”与“重置”操作
- 支持在文档中把未打卡写为正式记录，语法使用行末 `❌`
- 保证同一天同一习惯最终最多只有一条正式记录
- 支持左键与右键各自清晰分工，不和现有主打卡流程冲突
- 补打卡时按日期顺序插入到正确位置，而不是总追加到末尾
- 在习惯列表卡片中补充频率与下次打卡信息
- 保持旧习惯记录向后兼容

## 非目标

- 不在习惯列表主操作按钮中新增“未打卡”入口
- 不为“未打卡”引入独立的统计卡片
- 不改动现有习惯定义语法
- 不对历史文档做批量迁移
- 不支持同一天同时保留“已打卡”和“未打卡”两条记录
- 不把频率与下次打卡信息提升为卡片主标题级信息

## 决策

采用“正式记录状态显式化”方案：

- `CheckInRecord` 增加 `status: 'completed' | 'missed'`
- 文档中的行末 `❌` 记录作为正式 habit record 参与解析与日志展示
- 历史没有状态字段的记录默认视为 `completed`

在交互上采用“左键负责正向打卡，右键负责未打卡/重置”的拆分：

- 左键：
  - 空白日执行正常打卡
  - 未打卡日第一次左键只删除 `❌` 记录，回到空白态
  - 删除后再次左键，才执行正常打卡
- 右键：
  - 空白日只显示 `未打卡`
  - 已打卡日只显示 `重置`
  - 未打卡日只显示 `重置`

在习惯列表卡片上采用“底部左侧一行辅助信息”的布局：

- 显示内容为“频率 + 状态口吻”
- 位置位于卡片底部左侧，避开右侧操作区
- 今天需要打卡时只做轻量区分，不做强提醒样式

## 数据模型

在现有 `CheckInRecord` 上增加显式状态字段：

```typescript
interface CheckInRecord {
  content: string
  date: string
  completedAt?: string
  docId: string
  blockId: string
  currentValue?: number
  targetValue?: number
  unit?: string
  habitId: string
  status?: 'completed' | 'missed'
}
```

### 字段语义

- `completed`
  - 表示该记录是正常打卡记录
  - 二元型表示当天已完成
  - 计数型表示当天某次累计值记录，是否达标仍由现有计数逻辑判断
- `missed`
  - 表示该记录是显式未打卡记录
  - 该状态优先级高于计数型未达标的 `partial` 展示
  - 不计入任何“完成”相关统计

### 兼容策略

- 历史记录未写出 `status` 时，解析后默认视为 `completed`
- 运行时逻辑禁止同一天同一习惯同时保留 `completed` 与 `missed`

## Markdown 语法

### 已打卡

沿用现有语法：

```markdown
阅读 📅2026-05-11
喝水 3/8杯 📅2026-05-11

```

### 未打卡

新增行末 `❌` 语法：

```markdown
阅读 📅2026-05-11 ❌
喝水 📅2026-05-11 ❌

```

说明：

- `❌` 是正式记录语义，不是纯展示装饰
- 位置固定在记录行末尾，参考事项“已放弃”的 emoji 书写习惯
- 计数型未打卡不写 `current/target/unit`，只保留习惯名、日期和行末 `❌`
- 若启用了习惯打卡完成时间精度，则 `❌` 记录同样允许携带 `completedAt` 对应的日期时间格式，例如 `阅读 📅2026-05-11 21:30 ❌`
- 但若操作的不是“今日”记录，而是历史补打卡或历史未打卡，则时间精度一律收敛到日，只写 `YYYY-MM-DD`

## 交互设计

### 月历左键

- 空白日：
  - 二元型：执行正常打卡
  - 计数型：执行现有左键逻辑，创建或累加当日记录
- 未打卡日：
  - 第一次左键：只删除 `❌` 记录，让该日回到空白态
  - 第二次左键：才执行正常打卡
- 已打卡日：
  - 二元型保持现状，不重复新增
  - 计数型沿用当前主打卡逻辑，可继续累加或按现有规则更新当日记录

### 月历右键

- 空白日：只显示 `未打卡`
- 未打卡日：只显示 `重置`
- 已打卡日：只显示 `重置`
- 非有效日期不弹菜单：
  - 空白补位格
  - 习惯开始日前
  - 习惯结束日后

### 菜单动作语义

- `未打卡`
  - 为该日期创建一条 `status = 'missed'` 的正式记录
- `重置`
  - 删除该日期当前已有记录，让该日回到空白态
  - 若当前为已打卡，则删除当天对应的完成记录
  - 若当前为未打卡，则删除当天的 `❌` 记录

## 写入规则

### 单日唯一记录约束

同一天同一习惯最终只能保留一种状态：

- 空白 -> `未打卡`：新增 `missed`
- 空白 -> 左键打卡：新增 `completed`
- `missed` -> 左键第一次：删除 `missed`
- `missed` -> 右键 `重置`：删除 `missed`
- `completed` -> 右键 `重置`：删除当天完成记录

本次交互不支持在一次操作中把 `missed` 直接转换成 `completed`。用户若要从未打卡改成已打卡，必须先通过左键第一次或右键 `重置` 删除 `❌` 记录，再进行正常打卡。

### 补打卡插入位置

习惯记录在文档中的顺序应按日期升序排列。新增历史记录时，不能无脑追加到最后，必须寻找正确插入锚点。

规则如下：

1. 在当前习惯定义块下收集该习惯的所有 record
2. 按现有解析顺序定位目标日期前后的相邻记录
3. 若存在“目标日期之后的第一条记录”，则把新记录插入到它前面
4. 若不存在更晚记录，则插入到该习惯最后一条 record 后面

示例：

```markdown
阅读 🎯2026-05-01 🔄每天
阅读 📅2026-05-03
阅读 📅2026-05-05

```

若补打 `2026-05-04`，结果应为：

```markdown
阅读 🎯2026-05-01 🔄每天
阅读 📅2026-05-03
阅读 📅2026-05-04
阅读 📅2026-05-05

```

同理，若在 `2026-05-03` 与 `2026-05-05` 之间补一条 `阅读 📅2026-05-04 ❌`，也必须插在中间。

### 时间精度适用范围

习惯打卡完成时间精度设置只用于“今日记录”的写入。

规则如下：

1. 若操作日期等于当前日期：
   - 正常打卡可按全局时间精度写入 `completedAt`
   - 今日未打卡也可按全局时间精度写入，例如 `阅读 📅2026-05-11 21:30 ❌`
2. 若操作日期不是当前日期：
   - 无论是历史补打卡还是历史未打卡，都只写到日
   - 不写分钟或秒级时间

示例：

- 今日未打卡：`阅读 📅2026-05-11 21:30 ❌`
- 补 5 月 4 日未打卡：`阅读 📅2026-05-04 ❌`
- 补 5 月 4 日已打卡：`阅读 📅2026-05-04`

### 删除规则

- `重置` 和“未打卡日第一次左键”都走删除该记录行的路径
- 删除后不保留占位空行
- 删除后依赖现有增量刷新机制重新解析该文档

## 解析规则

habit record parser 需识别：

- 普通完成记录 -> `status = 'completed'`
- 行末含 `❌` 的记录 -> `status = 'missed'`

解析原则：

- 行末 `❌` 只在 habit record 语境下解释为未打卡标记
- 旧记录只要能解析出日期，默认视为 `completed`
- `missed` 记录仍需解析出 `date`、`completedAt`、`docId`、`blockId`、`habitId`
- 计数型 `missed` 记录不解析 `currentValue`、`targetValue`、`unit`

## 展示规则

### 列表卡片辅助信息

`HabitListItem` 在卡片底部左侧新增一行辅助信息，用于展示：

- 打卡频率
- 下次打卡时间或当前打卡状态

布局要求：

- 位于卡片底部左侧，即现有标题/连续天数/进度条下方的留白区域
- 与右侧文档按钮、主打卡按钮保持分区，不挤压操作命中区域
- 只占一行，超长时截断，不额外拉高卡片层级复杂度

文案风格采用“频率 + 状态口吻”，例如：

- `每天 · 今天该打卡了`
- `每周三次 · 明天再打卡`
- `每周一、三、五 · 今天无需打卡`

展示口径：

- 左侧第一段始终是频率摘要
- 第二段是面向用户的状态口吻，用于表达“今天该不该打卡”或“下一次何时需要打卡”

轻量区分规则：

- 今日需要打卡：使用比普通次级文案更醒目一点的颜色或字重
- 今日无需打卡：保持次级文案风格
- 不使用醒目的警告色块、角标或整卡高亮

文案优先级：

1. 若今天就是应打卡日，优先显示“今天该打卡了”
2. 若今天不是应打卡日，则显示“明天再打卡”或其他最近一次应打卡时间的自然语言描述
3. 若习惯已结束或已归档，则该文案应让位于结束/归档语义，不再突出“今天该打卡了”

### 月历状态

月历扩展为四种视觉状态：

- `completed`：显示 `✓`
- `missed`：显示 `✖`
- `partial`：仅计数型当天有完成记录但未达标时显示进度环
- `none`：显示空白圆点

优先级为：

```text
missed > completed > partial > none
```

这样可以确保显式未打卡一旦存在，就不会再显示计数型的进度环。

### 日志

`HabitRecordLog` 继续展示当月正式记录，包括：

- `completed`
- `missed`

展示规则：

- `missed` 记录直接展示带行末 `❌` 的内容
- 保持按日期倒序展示
- 点击仍跳转到对应文档块
- 记录被重置删除后，该日志项自然消失

## 统计规则

`missed` 记录不计入以下指标：

- `monthlyCheckins`
- `totalCheckins`
- `currentStreak`
- `longestStreak`
- 完成率分子

处理规则：

- 删除记录后，该日期按“无记录”处理
- 计数型习惯若当天存在 `missed`，当天直接视为未完成，不再按 `partial` 或达标逻辑参与月历表现
- 统计口径仍以“完成天数”计算，不因为 `missed` 增加新的分母定义

## 测试策略

### Parser

- 能识别 `阅读 📅2026-05-11 ❌`
- 历史 `阅读 📅2026-05-11` 默认解析为 `completed`
- 计数型 `喝水 📅2026-05-11 ❌` 不生成计数字段

### Domain

- `missed` 不计入完成次数与 streak
- 计数型同日存在 `missed` 时，月历应优先展示 `✖`
- 删除记录后，day state 回到空白态

### 写入

- 空白日右键 `未打卡` 会新增行末 `❌` 记录
- 已打卡日右键 `重置` 会删除当天完成记录
- 未打卡日右键 `重置` 会删除 `❌` 记录
- 历史补打 `2026-05-04` 时，会插到 `2026-05-03` 与 `2026-05-05` 之间
- 历史补打卡或历史未打卡时，即使全局精度为分/秒，也只写到日

### UI

- 未打卡日第一次左键只删除，不会顺手打卡
- 删除后第二次左键才执行正常打卡
- 日志中可见未打卡记录，删除后消失
- 列表卡片底部左侧显示“频率 + 状态口吻”辅助信息
- 今日应打卡与今日无需打卡有轻量视觉区分，但不喧宾夺主

## 风险与边界

- 若写入层无法稳定获取“目标日期之后的第一条记录”块 ID，补打卡插入顺序会错乱，因此实现时应优先复用解析后已有的 record 列表，而不是临时扫描原始文本
- 计数型习惯当前可能依赖“同日多次更新同一条记录”的逻辑，本次要避免把 `missed` 误接入累加路径
- `重置` 删除的是“该日期当前记录”，因此实现前需要统一“同一天唯一记录”的约束，避免残留多条旧数据时行为不确定
