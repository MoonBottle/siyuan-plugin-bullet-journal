# 习惯打卡完成时间精度设计

## 背景

当前习惯打卡记录只写入日期：

```markdown
喝水 📅2026-05-08
跑步 3/5公里 📅2026-05-08
```

这会带来两个问题：

- 用户无法区分同一天内具体完成于何时
- UI 展示与文档数据都只能看到“哪一天完成”，无法满足更细粒度的回顾需求

与此同时，习惯统计、周条、月历等核心逻辑本质上仍然是按“日”归档，不需要因为展示精度提升而改变统计口径。因此需要在不破坏现有按日模型的前提下，为习惯打卡补充可配置的完成时间精度。

## 目标

- 为习惯打卡增加全局“完成时间精度”设置
- 支持按设置写入到日、到分、到秒三种格式
- 习惯相关 UI 展示与写入格式保持一致
- 保持现有按日统计、月历、周条、完成率逻辑不变
- 向后兼容现有仅含日期的历史打卡记录

## 非目标

- 不为每个习惯单独设置时间精度
- 不修改 Todo 事项已有的时间精度模型
- 不对历史习惯记录做批量迁移或重写
- 不把时间精度加入周条、月历单元格的主视觉展示
- 不引入时区配置或 UTC 存储模型

## 决策

采用 **全局习惯打卡完成时间精度设置**，同时影响：

- 新增打卡记录写入 Markdown 的格式
- 习惯详情中相关时间展示的格式

设置值定义为：

```typescript
type HabitCheckInTimePrecision = 'day' | 'minute' | 'second'
```

默认值采用：

```typescript
'day'
```

默认保持当前行为，避免升级后无感知地改变现有用户文档格式。

## 数据模型

当前 `CheckInRecord` 中的 `date` 字段继续保留，语义仍然是“该记录归属于哪一天”，格式固定为：

```typescript
date: 'YYYY-MM-DD'
```

在此基础上新增原始完成时间字段：

```typescript
interface CheckInRecord {
  // ...existing fields
  completedAt?: string // 原始完成时间字符串，格式为 YYYY-MM-DD / YYYY-MM-DD HH:mm / YYYY-MM-DD HH:mm:ss
}
```

### 字段职责

- `date`
  - 用于按日统计
  - 用于周条、月历、完成率命中计算
  - 始终只保留日期部分
- `completedAt`
  - 用于保留文档中真实写入的完成时间精度
  - 用于 UI 展示
  - 缺失时表示旧数据或仅按日记录

### 为什么不直接复用 `date`

因为 `date` 目前已经被大量统计逻辑当作纯日期使用。若直接把它升级为日期时间字符串，会把以下逻辑全部带入隐式兼容成本：

- `record.date === currentDate`
- `startsWith('${viewMonth}-')`
- 月历格子命中
- 周条 7 天计算
- 频率达标判断

因此保留 `date` 的按日语义，新增 `completedAt` 承接精度扩展，是影响面最小、边界最清楚的做法。

## Markdown 格式

打卡记录中的日期标记从“仅日期”扩展为“日期或日期时间”。

### 二元型

```markdown
阅读 📅2026-05-08
阅读 📅2026-05-08 21:30
阅读 📅2026-05-08 21:30:45
```

### 计数型

```markdown
跑步 3/5公里 📅2026-05-08
跑步 3/5公里 📅2026-05-08 21:30
跑步 3/5公里 📅2026-05-08 21:30:45
```

## 设置设计

新增全局设置项：

- 分类：新增“习惯”设置区块
- 名称：`打卡完成时间精度`
- 可选值：
  - `到日`
  - `到分`
  - `到秒`

### 入口范围

- PC 设置弹窗提供该选项
- 移动端设置抽屉提供同样选项

### 不放在日历设置的原因

这个精度本质上约束的是“习惯打卡记录写入格式”，而不是日历渲染策略。若放在日历设置下，语义会偏移，也不利于未来继续承接习惯相关配置。

## 写入规则

习惯打卡时，按当前全局设置生成 `completedAt` 对应的 Markdown 时间片段。

### `day`

写入：

```markdown
📅2026-05-08
```

解析结果：

```typescript
{
  date: '2026-05-08',
  completedAt: '2026-05-08'
}
```

### `minute`

写入：

```markdown
📅2026-05-08 21:30
```

解析结果：

```typescript
{
  date: '2026-05-08',
  completedAt: '2026-05-08 21:30'
}
```

### `second`

写入：

```markdown
📅2026-05-08 21:30:45
```

解析结果：

```typescript
{
  date: '2026-05-08',
  completedAt: '2026-05-08 21:30:45'
}
```

### 生效范围

- 二元型首次打卡创建 record 时生效
- 计数型首次创建 record 时生效
- 计数型更新当日已有 record 时：
  - 保留原 record 的 `completedAt` 精度格式
  - 更新时间值时同步刷新为“本次操作时间”

这样同一天内持续累加的计数型记录，完成时间始终表示“最近一次更新该记录的时间”。

## 解析规则

习惯 record 解析器需同时兼容以下三种格式：

- `📅YYYY-MM-DD`
- `📅YYYY-MM-DD HH:mm`
- `📅YYYY-MM-DD HH:mm:ss`

也继续兼容旧别名 `@YYYY-MM-DD` 的日期解析逻辑，但不为 `@` 扩展分钟、秒级新写法。新能力统一只从 `📅` 标记进入。

### 解析结果规则

- `date` 始终取日期部分 `YYYY-MM-DD`
- `completedAt` 取完整原始字符串

示例：

| Markdown | `date` | `completedAt` |
|----------|--------|---------------|
| `喝水 📅2026-05-08` | `2026-05-08` | `2026-05-08` |
| `喝水 📅2026-05-08 08:30` | `2026-05-08` | `2026-05-08 08:30` |
| `喝水 📅2026-05-08 08:30:45` | `2026-05-08` | `2026-05-08 08:30:45` |

## 展示规则

### 展示范围

本次只要求习惯相关“明细型展示”遵循设置精度：

- 习惯详情中的月记录日志
- 后续若存在“最近一次打卡时间”“最后完成时间”类文案，也统一复用同一格式化函数

以下位置不展示时间：

- 周条
- 月历格子
- 统计卡片中的月完成率、总打卡次数、连续天数

原因是这些区域的主任务是传达“是否完成”和“累计统计”，加入时分秒只会增加噪音。

### 展示格式

日志中按当前设置展示：

- `day`: `5/8`
- `minute`: `5/8 21:30`
- `second`: `5/8 21:30:45`

### 历史记录兼容

旧记录若只包含日期：

- 展示 `5/8`
- 即使当前设置为 `minute` 或 `second`，也不补 `00:00` 或 `00:00:00`

这样可以避免制造伪精度，保证 UI 展示忠实反映文档中真实存在的数据。

## 统计与行为边界

时间精度升级后，以下逻辑保持不变：

- 今日是否已打卡判断
- `/dk` 命中今日 record 的更新逻辑
- 月历命中某日 record 的逻辑
- 周条最近 7 天完成情况
- 月完成率
- 总打卡次数
- 频率达标判断

这些逻辑继续只依赖 `date`，不依赖 `completedAt`。

## 交互影响

### 设置切换后的效果

- 新设置保存后，之后新产生的打卡记录按新精度写入
- 已存在记录不重写
- UI 展示优先使用记录自身实际解析出的 `completedAt`

因此切换设置后会出现“旧记录仍是到日，新记录是到分/到秒”的自然过渡，这是预期行为。

### 为什么不做历史迁移

因为：

- 功能尚未上线，没有必要为未发布格式引入批量迁移复杂度
- 历史记录本身并不存在更高精度信息，迁移只能补零，价值有限
- 当前目标是保证未来写入与展示一致，而不是伪造过去的时间细节

## 错误处理

### 解析失败

若 record 行日期时间格式非法：

- 不把非法时间解析为 `completedAt`
- `date` 无法提取时，整条 record 视为无效习惯 record

### 设置异常

若设置缺失或出现未知值：

- 回退到 `day`

## 测试范围

- `📅YYYY-MM-DD` 能正确解析
- `📅YYYY-MM-DD HH:mm` 能正确解析
- `📅YYYY-MM-DD HH:mm:ss` 能正确解析
- `date` 始终为纯日期
- `completedAt` 保留完整原始值

### 服务测试

- 二元型在三种精度下生成正确 Markdown
- 计数型首次创建在三种精度下生成正确 Markdown
- 计数型更新已有 record 时保留当前设置对应格式并更新时间

### 设置测试

- 默认值为 `day`
- 设置可保存与回读
- PC 与移动端设置组件都能正确回显和更新

### 展示测试

- 月记录日志在三种精度下展示正确
- 旧记录在高精度设置下不补零

### 回归测试

- 今日打卡判断不因 `completedAt` 引入回归
- 月完成率、总打卡次数、周条完成判断不变
- `/dk` 对习惯定义行和今日 record 的行为不变

建议按以下顺序实现：

1. 扩展设置模型与默认值
2. 增加 PC / Mobile 设置 UI
3. 扩展习惯 record 解析结构
4. 抽出习惯完成时间格式化工具
5. 更新打卡写入逻辑
6. 更新习惯日志展示
7. 补齐解析、服务、展示回归测试

## 验收标准

- 设置中可选择 `到日 / 到分 / 到秒`
- 新打卡记录按设置写入对应 Markdown 格式
- 习惯详情日志按记录自身真实精度展示
- 旧数据继续可解析
- 所有按日统计逻辑保持不变
