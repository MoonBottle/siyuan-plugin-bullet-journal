# 专注预算设计

日期：2026-05-13
状态：设计已确认，可进入实现计划
范围：事项级专注预算、斜杠命令入口、运行时解析，以及 V1 复盘模型

## 1. 背景

当前项目的番茄钟执行层已经比较完整，已具备：

- 从事项上下文开始专注
- 倒计时与正计时
- 暂停、继续、取消、完成
- 休息流程与自动延长
- 状态持久化与恢复
- 统计与历史记录

当前缺少的是番茄工作法中的“计划”部分。系统可以记录已经发生了多少专注，但还不能在执行前表达“原本预计要投入多少专注”。这会导致 GTD 风格的闭环缺失：

1. 先估算工作量
2. 再执行专注
3. 对比预计与实际
4. 持续修正计划质量

本设计在不改变插件“文档标记即数据源”原则的前提下，为事项补充专注预算能力。

## 2. 目标

V1 目标：

1. 允许每个事项在文档标记中存储一个预计专注预算。
2. 支持两种互斥的预计方式：
   - 预计番茄数
   - 预计专注时长
3. 提供基于斜杠命令的预算编辑入口。
4. 在现有番茄与事项流程中展示“预计 vs 实际”。
5. 基于事项预算与番茄记录，提供一个最小可用的每日复盘摘要。

## 3. 非目标

V1 明确不包含：

- 单独持久化“今日承诺清单”
- 计划内 / 计划外番茄分类
- 中断分类体系
- 每四个番茄后的长休息节奏逻辑
- 周复盘流程
- 在日历、甘特、工作台等所有界面全量铺开预计信息
- 自定义“1 个番茄等于多少分钟”的预计换算比例

## 4. 产品决策

### 4.1 预计粒度

预计信息存储在事项层。

原因：

- 当前番茄执行天然是围绕事项展开的。
- 事项是现有“项目 / 任务 / 事项”模型里最小的可执行单元。
- 任务层的预计汇总可以后续从事项层派生，不必先做双层计划。

### 4.2 每个事项只能有一个预计

每个事项在任意时刻只能有一个专注预算。

允许的模式：

- 番茄数
- 专注时长

不允许：

- 同一事项同时保存两种预计模式

### 4.3 文档标记仍是唯一真源

预计信息以内联标记的形式写在事项主行，不写入子块，也不写入块属性。

原因：

- 预计属于计划元信息，应该和事项文本一起可见
- 用户应当能手写和直接修改
- 这与现有插件依赖文档标记驱动解析的模式一致

### 4.4 固定归一化基准

在比较与汇总计算中，统一使用：

- `1 个番茄 = 25 分钟`

这个基准固定不变，不跟随用户当前默认专注时长设置变化。

原因：

- 历史预计必须保持语义稳定
- 否则用户修改默认专注时长后，历史预算的含义会被静默改写

## 5. 文档标记设计

### 5.1 规范写回格式

插件最终写回的规范格式只有以下几种：

- 番茄数：`🍅x3`
- 不足 1 小时的时长：`⏳45m`
- 含小时的时长：`⏳1h10m`

示例：

```text
- 设计优惠券核销流程 @2026-05-14 ⏳1h10m
- 修复提醒重复触发问题 @2026-05-14 🍅x3
```

### 5.2 兼容输入格式

解析器应兼容以下用户手写形式：

- `🍅x3`
- `🍅3`
- `⏳45m`
- `⏳70m`
- `⏳1h`
- `⏳1h10m`

写回时统一规范化：

- `🍅3 -> 🍅x3`
- `⏳70m -> ⏳1h10m`

### 5.3 冲突处理

如果同一事项行同时出现多个预计标记（无论是同类型还是不同类型）：

```text
- 示例事项 @2026-05-14 ⏳1h 🍅x3
- 示例事项 @2026-05-14 🍅x2 🍅x3
```

则：

1. 解析不能导致事项失效
2. 按从左到右顺序取第一个合法预计标记作为有效值
3. 后续预计标记在运行时忽略
4. 用户下次通过预计编辑器保存时，写回结果会只保留一个规范化标记

## 6. 运行时模型

V1 在事项层新增一个运行时概念结构：

```ts
interface FocusPlan {
  type: 'duration' | 'pomodoro'
  rawValue: number
  normalizedMinutes: number
  sourceText: string
  ignoredSourceTexts?: string[]
}
```

语义：

- `type='pomodoro'` 且 `rawValue=3`，表示 `normalizedMinutes=75`
- `type='duration'` 且 `rawValue=70`，表示 `normalizedMinutes=70`
- `sourceText` 保留最终生效的原始标记文本，便于调试和 UI 派生展示
- `ignoredSourceTexts` 可选记录被忽略的后续标记，便于调试

这个结构只存在于运行时 / 解析结果中，不需要脱离文档标记单独持久化。

## 7. 斜杠命令设计

### 7.1 新增内置命令

新增一个内置斜杠动作：

- `/focusplan`
- `/yj`

含义：

- 为当前事项设置或清除预计专注预算

### 7.2 触发行为

该命令仅对有效事项块生效。

流程：

1. 用户触发 `/focusplan` 或 `/yj`
2. 插件解析当前块是否为事项
3. 如果当前块不是有效事项：
   - 不打开编辑器
   - 直接提示“当前块不是有效事项”
4. 如果是有效事项：
   - 打开轻量预算弹框
   - 若该事项已有预计，则自动回填

这一模式应与现有 `/focus`、`/date`、`/setReminder`、`/setPriority` 的交互风格保持一致。

### 7.3 预计编辑弹框

弹框结构：

1. 顶部模式选择：
   - `预计时长`
   - `预计番茄`
2. 时长模式：
   - 小时输入
   - 分钟输入
3. 番茄模式：
   - 单个数字输入
4. 底部操作：
   - 确定
   - 取消
   - 清除预计（仅当当前事项已有预计时显示）

### 7.4 弹框规则

时长模式：

- 总时长必须大于 0
- 分钟输入会被归一化为合法总时长
- 写回采用规范格式：
  - `45m`
  - `1h10m`

番茄模式：

- 必须为正整数
- 写回采用规范格式 `🍅xN`

清除操作：

- 移除当前事项行上的所有 `⏳...` 与 `🍅x...` 标记

## 8. 写回规则

预计信息写回当前事项主行。

更新逻辑必须：

1. 先解析当前行
2. 移除当前行已有的预计标记
3. 若用户确认保存，则追加新的规范化标记
4. 保留其他元信息不变：
   - 日期
   - 提醒
   - 重复规则
   - 优先级
   - 状态标记
   - 标签与正文内容

结果应表现为：

- 预计编辑是一次“局部元信息重写”
- 不是自由文本整体替换
- 对历史上存在多个预计标记的事项，保存后会自动收敛为单个规范化标记

## 9. V1 显示范围

V1 只在以下 5 个位置展示预计专注信息。

### 9.1 事项详情弹窗

显示：

- 预计专注
- 实际专注
- 偏差

这里是信息最密集、最适合做复盘的主界面。

### 9.2 待办事项卡片

只做轻展示：

- `预计 3 🍅`
- `预计 1h10m`

若已有实际专注，可补一个紧凑进度形式：

- `50m / 1h10m`
- `2 / 3 🍅`

### 9.3 番茄开始弹框

当事项已有预计时，开始专注前展示：

- 当前预计
- 可选地展示该事项已累计的实际专注

让用户在开始前知道这轮专注处于该事项整体进度的哪个位置。

### 9.4 专注中界面

展示当前事项的轻量“实际 vs 预计”提示：

- `累计 50m / 1h10m`
- `累计 2 / 3 🍅`

### 9.5 专注统计视图

在现有统计视图中新增一个“预计复盘”摘要区：

- 今日有预计的事项数
- 今日预计总专注
- 今日实际总专注
- 超支 / 低于预计 / 未完成事项数量

## 10. 复盘模型

### 10.1 比较单位

V1 中所有“预计 vs 实际”的比较统一按分钟计算。

来源：

- 预计：
  - 时长预计 -> 直接转分钟
  - 番茄预计 -> `数量 * 25`
- 实际：
  - 该事项关联的番茄记录实际分钟数总和

### 10.2 复盘状态

对于带预计的事项，V1 赋予以下状态之一：

- `not-started`：有预计，但还没有任何实际专注
- `in-progress`：有预计，有实际专注，但事项未完成
- `matched`：事项已完成，且实际与预计接近
- `overrun`：事项已完成，且实际明显超过预计
- `underrun`：事项已完成，且实际明显低于预计

### 10.3 匹配阈值

使用固定阈值：

- 如果 `abs(actualMinutes - estimatedMinutes) <= 25`，则归类为 `matched`

原因：

- 一个标准番茄可以视为合理的计划波动区间
- 这样可以避免系统把轻微偏差过度分类

## 11. 每日摘要模型

V1 的每日摘要是推导结果，不额外持久化。

建议输出：

- 今日预计总分钟数
- 今日实际总分钟数
- 今日已完成且有预计的事项中：
  - matched 数量
  - overrun 数量
  - underrun 数量
- 今日仍未完成但已有预计的事项列表

这样可以最小化回答以下问题：

- 今天原本计划投入多少专注？
- 实际投入了多少？
- 哪些事项明显低估？
- 哪些事项还没做完？

## 12. 解析与聚合说明

### 12.1 多日期事项

项目当前已经存在多日期事项的视图派生和去重逻辑。

V1 规则：

- 专注预计属于事项本体，而不是属于每个派生日期副本
- 实际专注聚合继续沿用现有事项 / 番茄关联规则
- 每日摘要中必须避免同一底层事项预计被重复累计

### 12.2 历史番茄记录

V1 不要求修改现有番茄记录格式。

预计信息是计划元数据，实际专注仍完全来自现有 `🍅` 记录体系。

## 13. V1 范围清单

包含：

- 事项级专注预计标记
- 斜杠命令入口
- 预计编辑弹框
- 解析支持
- 规范写回
- 在限定界面展示预计信息
- 事项级“预计 vs 实际”复盘
- 最小每日复盘摘要

不包含：

- 今日计划持久化
- 中断原因记录
- 计划内 / 计划外分类
- 长休息节奏
- 周复盘流程
- 全量 UI 铺开

## 14. V2 方向

在 V1 稳定后，V2 可继续补全为更完整的工作流：

1. 显式的今日承诺清单
2. 计划内 / 计划外番茄区分
3. 中断记录与分类
4. 长休息节奏跟踪
5. 周复盘与长期低估分析

这些能力刻意延后，是为了让 V1 先聚焦解决“当前番茄系统没有计划层”的问题。

## 15. 实现风险

1. 行内标记重写必须避免破坏同一行上的其他元信息
2. 多个预计标记并存时，解析必须容错，不能导致事项整体识别失败
3. 多日期事项聚合必须避免预计总量重复计算
4. V1 显示范围必须保持克制，避免在整个插件引发大面积 UI 扰动

## 16. 建议结论

V1 应作为一次聚焦增强，把当前番茄模块从“记录做了多少”升级为“先计划，再执行，再复盘”。

关键路径是：

1. 事项预计标记
2. 斜杠命令编辑器
3. 解析与归一化
4. 限定界面展示
5. 预计 vs 实际复盘摘要

这是当前阶段最小但有价值的一步，既能补上计划层，又不会把系统直接推成一个重型日程规划器。
