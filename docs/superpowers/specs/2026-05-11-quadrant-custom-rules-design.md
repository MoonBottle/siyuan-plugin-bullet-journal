# 四象限自定义规则设计

日期：2026-05-11

## 背景

当前桌面端四象限视图和工作台相关象限组件都依赖固定映射：

- 第一象限：`priority = high`
- 第二象限：`priority = medium`
- 第三象限：`priority = low`
- 第四象限：`priority = none`

这套行为简单稳定，但无法支持类似滴答清单的“编辑象限”能力。用户希望每个象限都能单独配置规则，并且首版只实现“前两层规则链”，即只支持：

1. 第一层：优先级
2. 第二层：日期

同时，新的象限配置不应混入通用设置，而应以独立 JSON 文件持久化。工作台中的四象限视图和所有象限相关小组件也要与主四象限视图共用同一套配置与求值结果。

## 目标

本次设计交付以下能力：

1. 四个象限分别支持“编辑象限”弹框配置
2. 每个象限的规则链只支持两层：`优先级 -> 日期`
3. 四象限配置单独保存在插件数据目录中的 JSON 文件
4. 桌面端 `QuadrantTab`、工作台四象限视图、工作台象限摘要小组件统一适配新配置
5. 当启用自定义规则后，禁用四象限相关视图中的跨象限拖拽
6. 在编辑弹框左下角提供“恢复默认”按钮，一键把四个象限恢复为当前默认优先级行为

## 非目标

本次不做以下内容：

1. 不支持第三、第四层规则链
2. 不支持自由变更规则字段顺序
3. 不支持标签、清单、任务类型等字段
4. 不支持在工作台组件里覆盖全局象限配置
5. 不支持跨象限拖拽自动回写日期等复合字段
6. 不支持将未命中任务自动落入兜底象限

## 用户体验

### 主视图

桌面端四象限仍保持 2x2 网格，但每个象限不再由代码写死为固定优先级过滤器，而是由配置驱动：

- 标题来自该象限配置
- 列表内容来自该象限规则求值结果
- 计数来自该象限命中后的唯一归属结果

每个象限提供“编辑象限”入口，打开后展示与滴答清单相近的弹框布局。首版只暴露前两层规则链所需字段。

### 编辑象限弹框

每个象限都有自己的“编辑象限”弹框。弹框包含：

- 标题：`编辑象限`
- 当前象限标题编辑输入框
- 第一层规则：优先级多选
- 第二层规则：日期多选
- 底部规则摘要示例
- 右下角：`保存`、`取消`
- 左下角：`恢复默认`

其中“恢复默认”不是只重置当前象限，而是将四个象限整体恢复为默认配置：

- 第一象限：`priority = high`
- 第二象限：`priority = medium`
- 第三象限：`priority = low`
- 第四象限：`priority = none`

恢复默认后应立即更新独立配置文件，并使主四象限视图与工作台象限组件全部回到当前默认行为。

### 拖拽行为

自定义规则开启后，拖拽不再有稳定写回语义，因为目标象限可能由“优先级 + 日期”共同定义。为避免产生不可预测的数据修改：

- 主四象限视图禁用跨象限拖拽
- 工作台中的四象限视图和象限摘要组件同样禁用拖拽
- 视觉上不再显示可拖拽态
- 不再调用 `updateBlockPriority`

首版采用统一策略：只要四象限走新配置模型，就全部禁用跨象限拖拽，而不是区分“当前象限是否仍是单规则优先级配置”。

## 配置模型

四象限配置不存入 `settingsStore` 主设置对象，而是单独保存为一个 JSON 文件。

建议文件路径：

`/data/storage/petal/siyuan-plugin-bullet-journal/quadrant-config.json`

建议结构：

```ts
type QuadrantPriorityValue = 'high' | 'medium' | 'low' | 'none';
type QuadrantDateValue = 'overdue' | 'today' | 'tomorrow' | 'undated';

type QuadrantRuleChain = {
  priority?: QuadrantPriorityValue[];
  date?: QuadrantDateValue[];
};

type QuadrantPanelConfig = {
  id: 'q1' | 'q2' | 'q3' | 'q4';
  title: string;
  rules: QuadrantRuleChain;
};

type QuadrantConfigFile = {
  version: 1;
  panels: QuadrantPanelConfig[];
};
```

默认配置：

```json
{
  "version": 1,
  "panels": [
    { "id": "q1", "title": "重要且紧急", "rules": { "priority": ["high"] } },
    { "id": "q2", "title": "重要不紧急", "rules": { "priority": ["medium"] } },
    { "id": "q3", "title": "紧急不重要", "rules": { "priority": ["low"] } },
    { "id": "q4", "title": "不重要不紧急", "rules": { "priority": ["none"] } }
  ]
}
```

这里刻意不使用更抽象的“任意规则数组”结构，因为本期只支持固定两层链路。结构直接表达业务边界，后续若要扩展第三层规则链，再增量扩展模型即可。

## 存储与服务分层

新增一个专用的四象限配置服务，例如：

- `src/services/quadrantConfigService.ts`

职责：

1. 读取独立配置文件
2. 配置不存在时生成默认配置
3. 配置结构不完整时做回填和规范化
4. 保存配置文件
5. 对外提供“恢复默认配置”方法

建议该服务只负责文件读写、默认值回填和结构校验，不承担 UI 状态管理。界面层按需读取，或由专门的 store/composable 承载内存态。

如果需要减少重复读取，可以再增加一个轻量内存层，例如：

- `src/stores/quadrantConfigStore.ts`

该 store 只服务四象限配置，不合并到 `settingsStore`。

职责：

1. 持有已加载的四象限配置
2. 负责调用 `quadrantConfigService`
3. 向 `QuadrantTab` 和工作台组件暴露响应式配置
4. 统一触发配置更新后的界面刷新

## 规则求值

### 支持字段

本期仅支持两类规则：

1. 优先级
2. 日期

优先级候选值：

- `high`
- `medium`
- `low`
- `none`

日期候选值：

- `overdue`
- `today`
- `tomorrow`
- `undated`

### 求值语义

每个象限的规则是一个固定顺序的 `AND` 链：

1. 如果配置了 `priority`，事项优先级必须命中
2. 如果配置了 `date`，事项日期状态必须命中
3. 两者都命中时，事项才命中该象限

日期状态建议统一定义为：

- `overdue`：日期早于今天，且事项未完成
- `today`：日期为今天
- `tomorrow`：日期为明天
- `undated`：事项无日期

### 归属策略

在自定义配置下，同一事项可能命中多个象限，也可能一个都不命中。UI 上不能让一个事项同时出现在多个象限，因此需要统一归属规则。

采用固定顺序归属：

1. 按 `q1 -> q2 -> q3 -> q4` 的顺序判断
2. 事项归属到第一个命中的象限
3. 一旦归属成功，不再参与后续象限渲染
4. 未命中任何象限的事项不显示在四象限视图中

此策略同样适用于工作台所有象限相关组件，确保各处结果一致。

## 组件与模块变更

### 主视图

`src/tabs/QuadrantTab.vue`

改造为配置驱动视图：

1. 不再直接依赖 `QUADRANT_DEFINITIONS` 作为最终筛选依据
2. 从四象限配置服务或专用 store 读取四个象限配置
3. 根据配置渲染面板标题
4. 根据统一求值结果渲染每个面板的数据与计数
5. 提供每个象限的“编辑象限”入口
6. 在新配置模型下禁用拖拽

`src/utils/quadrant.ts`

不再只承担“固定优先级定义”角色。可以重构为：

- 默认四象限配置常量
- 与旧工作台象限 key 的映射工具
- 通用标题/默认值辅助函数

### 编辑弹框

新增独立弹框组件，例如：

- `src/components/quadrant/QuadrantRuleDialog.vue`

职责：

1. 编辑单个象限的标题与规则
2. 实时展示规则摘要
3. 保存当前象限配置
4. 提供“恢复默认”入口

“恢复默认”按钮触发的是全局重置逻辑：

1. 弹出确认
2. 用默认四象限配置覆盖当前配置文件
3. 刷新主视图与工作台所有象限组件

这里必须明确：虽然按钮位于当前象限弹框左下角，但行为是“恢复全部四个象限为默认配置”，不是“只恢复当前象限”。

### 求值器

新增专用求值模块，例如：

- `src/utils/quadrantEvaluator.ts`

建议职责拆分：

1. 将 `Item` 映射为可供规则判断的归一化上下文
2. 计算事项日期状态
3. 判断事项是否命中某个象限
4. 计算一组事项在四个象限中的唯一归属结果

对外可以暴露两类 API：

1. 单事项是否命中单象限
2. 批量事项的四象限归属结果

前者便于单测，后者便于视图层复用，避免每个组件重复扫描项目列表。

## 工作台适配

工作台不仅包含嵌入式四象限视图，还包含象限相关小组件。本次全部切换到同一套独立配置与求值结果。

### 嵌入式四象限视图

`src/components/workbench/view/WorkbenchViewHost.vue`

当前直接嵌入 `QuadrantTab`。这个部分不需要额外维护配置副本，只要 `QuadrantTab` 已经改为读取独立配置文件，工作台里的四象限视图就会自动同步。

### 象限摘要小组件

`src/components/workbench/widgets/QuadrantSummaryWidget.vue`

需要从“固定 `high/medium/low/none` 过滤”改为：

1. 读取 widget 所指向的象限 ID，例如 `q1`
2. 读取独立四象限配置
3. 使用统一求值器计算该象限的唯一归属事项
4. 使用象限配置里的标题渲染组件标题

### 小组件配置弹框

`src/components/workbench/dialogs/QuadrantWidgetConfigDialog.vue`

现状使用 `QUADRANT_DEFINITIONS` 生成下拉项。改造后需要：

1. 读取四象限配置文件
2. 下拉项展示 `q1 / q2 / q3 / q4` 及其实时标题
3. 选择结果保存为象限 ID，而不是旧的 `high/medium/low/none`

### workbench 类型

`src/types/workbench.ts`

现状：

```ts
export type WorkbenchQuadrantKey = 'high' | 'medium' | 'low' | 'none';
```

需要改为新的稳定标识：

```ts
export type WorkbenchQuadrantKey = 'q1' | 'q2' | 'q3' | 'q4';
```

这样工作台配置与象限标题解耦。即使用户把标题改成任意文案，工作台引用关系也不会失效。

### widget 注册默认值

`src/workbench/widgetRegistry.ts`

默认配置应从旧的：

- `quadrant: 'high'`

改为：

- `quadrant: 'q1'`

并同步更新打开配置弹框时的默认处理逻辑。

## 兼容与迁移

### 四象限配置文件迁移

老用户不存在 `quadrant-config.json` 时：

1. 读取默认四象限配置
2. 保存默认文件
3. 主视图与工作台按默认行为渲染

若文件存在但结构缺失或版本不匹配：

1. 以当前 `version: 1` 进行规范化
2. 缺失的 panel 用默认值补齐
3. 非法字段丢弃

### 工作台旧配置迁移

旧工作台 widget 若仍保存：

- `high`
- `medium`
- `low`
- `none`

则在读取阶段映射为：

- `high -> q1`
- `medium -> q2`
- `low -> q3`
- `none -> q4`

保存时统一写回新值，避免长期维持双格式。

## 风险与取舍

### 风险一：同一事项命中多个象限

允许配置重叠会带来“为什么事项没出现在另一个象限”的理解成本。首版不禁止重叠，而是通过固定顺序归属降低实现复杂度。这是有意识的取舍。

### 风险二：规则配置更新后的刷新一致性

主视图、工作台视图、工作台小组件都依赖同一个配置文件。若刷新链路不统一，容易出现部分区域已更新、部分区域仍显示旧结果。因此建议统一通过四象限配置服务或专用 store 触发刷新，而不是在多个组件里各自读取文件。

### 风险三：拖拽能力回退

禁用拖拽会比当前功能弱，但这是为了避免在“优先级 + 日期”组合下错误修改任务属性。首版先保证规则视图正确，再评估后续是否提供受限拖拽策略。

## 实施摘要

本次改造的核心不是给现有固定面板加几个过滤条件，而是把四象限从“固定优先级视图”升级为“配置驱动的矩阵视图”。

落地策略：

1. 每个象限独立配置标题和前两层规则链
2. 配置单独持久化到 `quadrant-config.json`
3. 用统一求值器产出四象限唯一归属结果
4. 主四象限视图和工作台所有象限组件全部消费这份结果
5. 在编辑弹框左下角提供“恢复默认”，一键恢复四个象限为当前优先级默认行为

这样可以在不引入完整通用规则引擎的前提下，先把“滴答式象限配置”的核心体验做出来，并为后续扩展第三、第四层规则链保留清晰边界。
