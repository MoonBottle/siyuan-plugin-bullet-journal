Block Writer 当前架构图

这份文档梳理 src/utils/blockWriter 当前的入口路由、实际写入链路、典型调用栈，以及后续重构最值得动的点。 结论先行：现在的 blockWriter 已经不是单一 writer，而是一个“按 patch 组合和上下文动态分流”的 dispatcher， 其中 date、slash + status、Protyle DOM、API/kramdown 四类实现并存。

入口: writeBlock() DOM/Transaction 路径 Kramdown/API 路径 组合 patch 特判 重复逻辑开始增多

总体路由图

一句话总结： writeBlock() 现在承担了“策略分发器”的角色，但策略本身仍然散落在 datePatchWriter.ts、statusPatchWriter.ts、protyleTransport.ts、 markdownWriter.ts 和 apiTransport.ts 里。

模块职责表

模块 当前职责 观察
index.ts 统一入口，按 patch 组合和上下文分流。 核心路由规则集中在这里，但形式已经是累加式 if/else。
protyleTransport.ts 处理适合 DOM 局部编辑的 patch，直接提交 transaction。 适合细粒度 patch，但对组合 patch 没有统一批处理模型。
statusPatchWriter.ts 处理 removeSlashCommand + setStatus 的组合写入。 已经复制了 range 定位、DOM 草稿、updated attr、transaction 提交逻辑。
datePatchWriter.ts 处理 addDate，兼顾 source 解析、markdown 生成、writer fallback。 职责最重，也是目前最像“子系统”的模块。
markdownWriter.ts 把完整 markdown 渲染回当前 block，必要时降级到 API。 和 statusPatchWriter / protyleTransport 都有 transaction 提交重复。
apiTransport.ts API 写入与插入，依赖 resolver 和 markdown -> DOM 序列化。 简单直接，但不知道 slash / selection / 当前 DOM 状态。
blockTargetResolver.ts 解析 status patch 应该落到 paragraph 还是 task list item。 replaceMode 已暴露但当前基本未发挥作用。
kramdownModifier.ts 纯 markdown patch 引擎。 这是目前最干净、最容易抽离测试的层。

当前路由矩阵

输入 优先路径 失败后的后备路径
[removeSlashCommand, setStatus] writeStatusWithSlashCleanup() 先 writeViaProtyle(removeSlash)，再等待 transaction flush，最后递归 writeBlock(setStatus)
[removeSlashCommand, addDate] writeDatePatchWithSlashCleanup() 先删 slash，再等待 flush，最后走 writeDatePatch()
单独 addDate writeDatePatch() 由 createProtyleMarkdownWriter() 决定 inline 或 API
有 protyle + nodeElement 的普通 patch writeViaProtyle() 若不是 slash-only 场景，则继续落到 writeViaApi()
没有 Protyle 上下文 writeViaApi() 无
insertBlockAfter* insertViaApi() 无

关键调用栈

/fq 非任务块

现在已经收敛为单次批量写入。

  1. slashCommands.ts 组装 writeContext
  2. writeBlock([removeSlashCommand, setStatus])
  3. writeStatusWithSlashCleanup()
  4. 克隆当前 DOM，按 slash range 删除 /fq
  5. blockElementToMarkdownContent() 取草稿 markdown
  6. applyBlockPatch(setStatus) 生成新 markdown
  7. renderMarkdownIntoBlockEditable() 回写当前块
  8. protyle.transaction(update, undo)

/fq 任务列表块

和上面同链，但提交目标变成 list item。

  1. slashCommands.ts 带上 listItemBlockId
  2. writeBlock([removeSlashCommand, setStatus])
  3. writeStatusWithSlashCleanup()
  4. 定位外层 NodeListItem 作为 targetElement
  5. 删除 slash，生成草稿 markdown
  6. applyBlockPatch() 为任务行补 ❌
  7. 回写 list item DOM
  8. 按 listItemBlockId 提交单次 transaction

/jt 或 slash + 日期

这是另一条组合分支。

  1. slash handler 调 writeBlock([removeSlashCommand, addDate])
  2. writeDatePatchWithSlashCleanup()
  3. 先从当前 DOM 草稿里删 slash
  4. prepareDatePatchWriteFromSource() 生成最终 markdown
  5. writeMarkdownToCurrentBlock() 提交 transaction
  6. 若无法同块完成，再回退到 writeDatePatch() / API

普通状态写入

不带 slash 的 status 仍然是双实现。

  1. writeBlock(setStatus)
  2. 若有 Protyle 上下文，先走 writeViaProtyle()
  3. 任务列表完成态可能直接 toggle checkbox DOM
  4. 否则 handleRenderedMarkdownPatchViaDOM()
  5. 还不行就 writeViaApi()
  6. resolveApiBlockTarget() 决定目标 block
  7. applyBlockPatch() + updateBlock()

调用栈对照

slashCommands.ts
  -> writeBlock(context, patches)
     -> [组合分支] writeStatusWithSlashCleanup()
        -> resolveSlashContext()
        -> clone targetElement and deleteSlashRangeText()
        -> blockElementToMarkdownContent()
        -> applyBlockPatch(splitKramdownBlock(markdown), patch)
        -> renderMarkdownIntoBlockEditable()
        -> commitProtyleUpdate()
           -> protyle.transaction(doOps, undoOps)

writeBlock(context, patch)
  -> [普通 Protyle 分支] writeViaProtyle()
     -> handleRemoveSlashViaDOM()
     -> handleSetStatusViaDOM()
     -> handleRenderedMarkdownPatchViaDOM()
  -> [API fallback] writeViaApi()
     -> resolveApiBlockTarget()
     -> getBlockKramdown()
     -> applyBlockPatch/applyBlockPatches()
     -> markdownToBlockDOM()
     -> updateBlock()

现在最明显的结构问题

  • 路由决策和 patch 实现耦合太深。 新增一个组合 patch，往往要同时改 writeBlock()、某个专用 writer、对应 slash handler 和测试。
  • DOM 事务提交逻辑重复。 formatUpdatedAttr()、transaction(do, undo)、oldHTML/newHTML 比较在多个文件里出现。
  • slash 上下文恢复逻辑重复。 resolveSlashContext()、getNodePath()、getNodeByPath() 在 datePatchWriter 和 statusPatchWriter 里各有一份。
  • datePatchWriter 过重。 它同时负责 block source 选择、日期表达式优化、跨父文档场景、同块 transaction 和 API fallback。
  • 调用方协议不一致。 /fq 已经是批量组合写入，/done 仍然是“先 removeSlash，再 setStatus”的两阶段流程。
  • API 路径对 DOM 结构认知不足。 它知道 markdown，却不知道当前编辑中的真实 DOM/selection 状态，遇到 slash 场景天然弱势。

可优化方向

  • P0抽出统一的 Protyle mutation session。 把 slash context、DOM 草稿、markdown 生成、updated attr、transaction commit 做成一个共享工具，替换 statusPatchWriter、datePatchWriter、markdownWriter 里的重复代码。
  • P0把 writeBlock 的组合分支改成策略注册表。 例如声明 matches(context, patches) 和 execute(context, patches)，避免继续堆 if/else。
  • P1统一 slash 命令的调用协议。 /done、/fq、未来类似命令都应优先产出批量 patch，而不是在 handler 里自己拆步骤。
  • P1收敛 block target 语义。 现在 listItemBlockId、resolveApiBlockTarget()、statusPatchWriter 都在决定“真实写入目标”，应该归一到一层能力模型里。
  • P1拆分 datePatchWriter。 可以按 source resolver、date expression transformer、writer adapter 三部分拆开。
  • P2减少 API DOM 写入。 对 list item 或结构复杂块，优先选择 markdown 或 transaction，避免 Md2BlockDOM() 生成新结构带来的 id/undo 风险。
  • P2加显式 trace。 给每次写入打 route、targetBlockId、patchTypes、writerType，后续排查会快很多。
  • P2测试按“patch 组合 x block 类型 x transport”建矩阵。 现在已有定向回归，但还不是完整的可枚举矩阵。

推荐的下一步重构顺序

  1. 先提取 protyleMutationSession，消除 range 恢复、DOM 草稿和 transaction 提交的重复代码。
  2. 再把 writeBlock() 的特判分支迁到注册表式策略层，让路由从 imperative if/else 变成 declarative rules。
  3. 随后统一 slash command 调用侧，让 /done 也升级到和 /fq 一样的批量写法。
  4. 最后拆 datePatchWriter，把当前最大的复杂度源头削成可单测的小模块。
如果只做一件事，优先做第 1 步。因为它既能减少重复，也能为后续 status/date/slash 的所有组合路径提供统一底座。
